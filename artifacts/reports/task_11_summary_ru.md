# TASK-11: средство обмена данными с ArduPilot через JSON и UDP

## Назначение задачи

TASK-11 предназначена для перехода от подготовительной заготовки TASK-10
к первому воспроизводимому уровню обмена данными между
`universal-mr-uav` и `ArduPilot JSON SITL`.

На данном этапе требовалось:

- реализовать кодирование расчетного пакета данных в строку `JSON`;
- реализовать разбор двоичного пакета выходов `ArduPilot`;
- реализовать транспортный уровень `UDP` в MATLAB;
- выполнить самопроверку без установленного `ArduPilot`;
- выполнить короткую попытку обмена с уже запущенным `ArduPilot SITL`,
  если среда готова;
- сохранить исходные журналы прогонов и инженерный отчет.

Задача не предусматривала:

- подтверждение устойчивого автоматического полета;
- летную валидацию;
- внедрение `MAVLink`;
- настройку режимов взведения, взлета, посадки и переключения режимов;
- перенос физики объекта управления в `Simulink`.

## Выполненные изменения

В пакете `src/+uav/+ardupilot/` добавлены или расширены следующие
средства:

- `encode_json_fdm_text.m` для кодирования пакета данных в строку
  `JSON`;
- `decode_sitl_output_packet.m` для разбора двоичного пакета
  `ArduPilot`;
- `json_udp_is_available.m` для проверки доступности `UDP` в MATLAB;
- `json_udp_open.m`, `json_udp_step.m`, `json_udp_close.m` для открытия,
  пошагового обмена и закрытия транспортного уровня;
- `make_sitl_start_command.m` для формирования рекомендуемой команды
  запуска `ArduPilot` в режиме `JSON`;
- `run_case_with_ardupilot_udp.m` для расчетного сценария обмена
  с использованием существующей математической модели движения.

Дополнительно созданы сценарии верхнего уровня:

- `scripts/run_ardupilot_json_udp_selfcheck.m`;
- `scripts/run_ardupilot_json_udp_handshake.m`.

Также добавлены новые модульные проверки для:

- кодирования строки `JSON`;
- разбора двоичных пакетов на 16 и 32 канала;
- проверки доступности `UDP`;
- формирования рекомендуемой команды запуска;
- сценария локальной самопроверки.

## Проведенные проверки

Локально выполнены следующие команды:

```matlab
scripts/bootstrap_project.m
runtests('tests')
scripts/run_ardupilot_json_udp_selfcheck.m
scripts/run_ardupilot_json_udp_handshake.m
```

По результатам запусков сохранены исходные журналы прогонов:

- `artifacts/logs/task_11_bootstrap.txt`;
- `artifacts/logs/task_11_runtests.txt`;
- `artifacts/logs/task_11_json_udp_selfcheck.txt`;
- `artifacts/logs/task_11_json_udp_handshake.txt`.

Все журналы сохранены как текстовые файлы UTF-8 и проходят проверку на
отсутствие скрытых и управляющих символов.

## Результат проверки UDP

Проверка `json_udp_is_available.m` показала:

- `UDP` доступен;
- используемый механизм — `udpport`.

Сценарий локальной самопроверки без `ArduPilot` подтвердил:

- корректность кодирования строки `JSON`;
- корректность разбора искусственно сформированного двоичного пакета;
- корректность преобразования длительности импульсов ШИМ в команды
  частоты вращения винтов;
- корректность локального обмена по `UDP`.

Следовательно, средство обмена на уровне MATLAB и `udpport`
работоспособно для внутренней проверки без внешнего программного
комплекса управления полетом.

## Готовность локальной среды

Проверка готовности локальной среды показала:

- локальная среда не готова к реальному `ArduPilot SITL`;
- `WSL` не обнаружен;
- `Python` доступен;
- `sim_vehicle.py` не обнаружен;
- локальный каталог `ArduPilot` не найден.

Итоговый логический признак готовности:

- `ready for real SITL = no`.

## Результат попытки обмена с ArduPilot

Сценарий `scripts/run_ardupilot_json_udp_handshake.m` выполнил короткую
попытку обмена с уже запущенным `ArduPilot JSON SITL`.

Фактический результат оказался следующим:

- двоичный пакет от `ArduPilot` не получен;
- локальная среда не готова к реальному `ArduPilot SITL`;
- рекомендуемая команда запуска сформирована;
- исходящая пробная строка `JSON` была сформирована и отправлена.

При этом необходимо отдельно зафиксировать:

- ответ на принятый двоичный пакет `ArduPilot` не проверен, так как
  входной пакет не был получен.

Таким образом, на данном этапе подтверждена только исходящая пробная
передача через `UDP`, но не подтвержден двусторонний обмен с работающим
`ArduPilot`.

## Результаты модульных проверок

Полная команда `runtests('tests')` дала следующий результат:

- всего проверок: `56`;
- пройдено: `56`;
- ошибок: `0`;
- незавершенных: `0`.

Модульные проверки подтвердили:

- наличие обязательных полей в строке `JSON`;
- корректный разбор двоичного пакета с `magic = 18458`;
- корректный разбор двоичного пакета с `magic = 29569`;
- устойчивую обработку короткого или пустого двоичного пакета без
  аварийного завершения;
- корректное преобразование ШИМ `1000 / 1500 / 2000` мкс в команды
  частоты вращения винтов через существующие функции;
- корректную проверку доступности `UDP`;
- корректное формирование рекомендуемой команды запуска
  `ArduPilot JSON SITL`;
- корректную работу сценария самопроверки.

## Ограничения текущего этапа

На текущем этапе сохраняются следующие ограничения:

- двусторонний обмен с работающим `ArduPilot SITL` не подтвержден;
- устойчивый автоматический полет не подтвержден;
- летная валидация не проводилась;
- режимы взведения, взлета, посадки и переключения режимов не
  согласованы;
- `MAVLink` не реализован;
- отсутствие принятого двоичного пакета честно фиксируется без подмены
  искусственными данными в сценарии реального обмена.

Следовательно, TASK-11 следует рассматривать как этап инженерной
проверки интерфейса и транспортного уровня, а не как завершенное
сопряжение с работающим внешним программным комплексом управления.

## Вывод

TASK-11 успешно довела подготовительную заготовку TASK-10 до первого
воспроизводимого уровня обмена данными по `JSON` и `UDP`.

Подтверждено:

- формирование исходящего пакета данных;
- корректный разбор двоичного пакета заданного формата;
- доступность и работоспособность `UDP` в MATLAB;
- воспроизводимая локальная самопроверка;
- корректное завершение сценария обмена при отсутствии принятого пакета
  от `ArduPilot`.

Не подтверждено:

- получение двоичного пакета от реального `ArduPilot`;
- ответная передача после принятого пакета;
- устойчивое длительное взаимодействие с работающим `ArduPilot SITL`.

## Следующий этап

Следующий этап работ должен включать:

1. подготовку рабочей среды `WSL`, если используется Windows;
2. получение локального checkout `ArduPilot`;
3. запуск `sim_vehicle.py` в режиме `JSON`;
4. подтверждение приема двоичного пакета от `ArduPilot`;
5. подтверждение ответной передачи `JSON` после принятого пакета;
6. согласование порядка двигателей и диапазонов ШИМ;
7. расширение проверок до длительного двустороннего программного
   моделирования перед переходом к более сложным режимам.
