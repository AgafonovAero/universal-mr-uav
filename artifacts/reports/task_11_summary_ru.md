# TASK-11: средство обмена данными с ArduPilot через JSON и UDP

## Назначение задачи

TASK-11 предназначен для перехода от подготовительного этапа TASK-10 к
первому воспроизводимому обмену данными между `universal-mr-uav` и
`ArduPilot JSON SITL`.

На данном этапе требовалось:

- реализовать кодирование расчетного пакета данных в строку `JSON`;
- реализовать разбор двоичного выходного пакета `ArduPilot`;
- реализовать транспортный уровень `UDP` в MATLAB;
- выполнить локальную самопроверку без установленного `ArduPilot`;
- выполнить краткую попытку обмена с реальным `ArduPilot SITL`, если
  среда готова;
- сохранить исходные журналы прогонов, отчет и документацию.

Задача не предусматривала:

- подтверждение устойчивого автоматического полета;
- реализацию `MAVLink`;
- настройку режимов взведения, взлета, посадки и переключения режимов;
- перенос физики объекта управления в `Simulink`;
- летную валидацию.

## Выполненные изменения

В пакете `src/+uav/+ardupilot/` добавлены или расширены следующие
компоненты:

- `default_json_config.m` — типовая конфигурация обмена по `JSON` и
  `UDP`, включая параметры транспортного уровня и форматы двоичного
  пакета;
- `encode_json_fdm_text.m` — кодирование расчетного пакета данных в
  строку `JSON` с завершающим переводом строки;
- `decode_sitl_output_packet.m` — разбор двоичного пакета `ArduPilot`
  для вариантов на 16 и 32 канала;
- `json_udp_is_available.m` — проверка доступности `UDP` в MATLAB;
- `json_udp_open.m`, `json_udp_step.m`, `json_udp_close.m` — открытие,
  один шаг обмена и закрытие транспортного уровня;
- `make_sitl_start_command.m` — формирование рекомендуемой команды
  запуска `ArduPilot` в режиме `JSON`;
- `run_case_with_ardupilot_udp.m` — расчетный сценарий обмена, который
  использует реальный двоичный пакет `ArduPilot`, если он получен.

Дополнительно созданы:

- `scripts/run_ardupilot_json_udp_selfcheck.m` — локальная самопроверка
  без установленного `ArduPilot`;
- `scripts/run_ardupilot_json_udp_handshake.m` — краткая попытка обмена
  с уже запущенным `ArduPilot SITL`;
- `docs/71_ardupilot_json_udp_ru.md` — профильный документ по TASK-11;
- новые модульные проверки для кодирования `JSON`, разбора двоичного
  пакета, доступности `UDP`, формирования команды запуска и сценария
  самопроверки;
- исходные журналы прогонов TASK-11.

Обновлены:

- `README.md`;
- `docs/70_ardupilot_json_sitl_ru.md`.

## Результат проверки UDP

Проверка `json_udp_is_available` показала, что в текущей версии MATLAB
доступен механизм `udpport`.

Локальная самопроверка без `ArduPilot` подтвердила:

- доступность `UDP` в MATLAB: да;
- используемый механизм: `udpport`;
- корректность кодирования расчетного пакета в строку `JSON`: да;
- корректность разбора искусственно сформированного двоичного пакета:
  да;
- корректность преобразования длительности импульсов ШИМ в команды
  частоты вращения винтов: да;
- двусторонний локальный обмен по `UDP` на проверочной датаграмме: да.

## Был ли доступен реальный ArduPilot SITL

Реальный `ArduPilot SITL` в текущей локальной среде недоступен.

Проверка готовности среды показала:

- `is_ready = false`;
- `has_wsl = false`;
- `has_python = true`;
- `has_sim_vehicle = false`;
- локальный каталог `ArduPilot` не обнаружен.

Следовательно, TASK-11 выполнялся в условиях отсутствия готовой
платформы для длительного реального обмена с `ArduPilot SITL`.

## Был ли получен двоичный пакет от ArduPilot

Краткая попытка обмена с реальным `ArduPilot SITL` завершилась штатно,
но двоичный пакет от `ArduPilot` не был получен.

Сценарий `scripts/run_ardupilot_json_udp_handshake.m` зафиксировал:

- готовность `UDP`: да;
- готовность локальной среды: нет;
- пакет от `ArduPilot` получен: нет;
- команда запуска для отдельного прогона сформирована: да.

Рекомендуемая команда запуска:

```text
sim_vehicle.py -v ArduCopter --model JSON:127.0.0.1 --console --map
```

## Был ли отправлен ответный JSON

Несмотря на отсутствие двоичного пакета от реального `ArduPilot`,
сценарий обмена сформировал и отправил строку `JSON`.

Итоговый статус из журнала handshake:

```text
Пакет от ArduPilot не получен; Строка JSON отправлена.
```

Это означает, что транспортный уровень и кодирование `JSON` способны
сформировать исходящий пакет, однако отсутствие встречного двоичного
пакета не позволяет считать сопряжение подтвержденным на уровне
реального внешнего комплекса управления.

## Ограничения текущего этапа

На текущем этапе сохраняются следующие ограничения:

- отсутствует подтвержденный обмен с работающим `ArduPilot SITL`;
- не подтвержден устойчивый автоматический полет;
- не согласованы команды взведения, взлета и переключения режимов;
- не выполнено длительное сопряжение на продолжительном расчетном
  прогоне;
- летная валидация не проводилась.

Следовательно, TASK-11 следует рассматривать как первый воспроизводимый
этап обмена данными, а не как завершенное средство сопряжения,
пригодное к штатной эксплуатации.

## Следующий этап

Следующий этап работ должен включать:

1. подготовку рабочей среды `WSL`;
2. получение локального checkout `ArduPilot`;
3. запуск `sim_vehicle.py` в режиме `JSON`;
4. подтверждение устойчивого двустороннего обмена по `UDP`;
5. согласование порядка двигателей, диапазонов ШИМ и семантики команд;
6. расширение расчетных сценариев с реальным ответом `ArduPilot`;
7. отдельную верификацию устойчивости до перехода к более сложным
   режимам и последующей летной валидации.
