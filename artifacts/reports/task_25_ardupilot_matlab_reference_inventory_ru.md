## Правила официального примера

### Открытие UDP

Официальный пример MATLAB backend:

- открывает UDP-сокет на локальном порту `9002`;
- не требует заранее знать удаленный порт ArduPilot;
- отвечает на адрес и порт последнего принятого пакета через
  `pnet(u,'gethost')`.

Следствие для нашего проекта:

- MATLAB backend обязан слушать `9002`;
- адрес и порт ответа должны браться из последнего валидного пакета SITL,
  а не из фиксированного target-конфига.

### Формат входного бинарного пакета

Официальный бинарный пакет ArduPilot JSON SITL имеет вид:

- `uint16 magic`
- `uint16 frame_rate`
- `uint32 frame_count`
- `uint16 pwm[16]`

При `SERVO_32_ENABLE = 1`:

- `magic = 29569`
- `pwm[32]`

Из пакета обязательно используются:

- `magic` для проверки протокола;
- `frame_rate` для вычисления шага физики;
- `frame_count` для обнаружения дублей, пропусков и перезапуска SITL;
- `pwm` как вход внешнего объекта управления.

### Вычисление шага интегрирования

В официальном `SITL_connector.m` шаг физики задается формулой:

`delta_t = min(1 / frame_rate, max_timestep)`

Следствие:

- внешний backend не должен жить на собственном фиксированном таймере;
- временной шаг должен задаваться пришедшим `frame_rate`;
- верхняя граница шага допустима как ограничение численной устойчивости.

### Когда формируется JSON

Официальный пример работает по строгой схеме lockstep:

1. принят валидный бинарный пакет ArduPilot;
2. выполнен ровно один шаг физики;
3. сформирован ровно один JSON-ответ;
4. JSON отправлен обратно на адрес отправителя.

JSON не должен отправляться сам по себе без нового принятого входного
кадра, если это не специально выделенный диагностический режим.

### Обязательные поля JSON

Согласно `libraries/SITL/examples/JSON/readme.md`, обязательны:

- `timestamp`
- `imu.gyro`
- `imu.accel_body`
- `position`
- `velocity`
- `attitude` или `quaternion`

Для TASK-25 выбран основной официальный вариант:

- `attitude = [roll pitch yaw]`

`quaternion` допускается только как отдельный дополнительный режим и не
используется одновременно без необходимости.

### Формирование accel_body в официальном MATLAB-примере

В `SIM_multicopter.m` используется следующая логика:

1. рассчитывается удельная сила в связанной системе координат;
2. она переводится в земную систему координат;
3. добавляется сила тяжести;
4. на земле вертикальная составляющая вниз ограничивается нулем;
5. затем обратно формируется показание акселерометра
   `accel_body = C_nb' * (accel_ned + [0 0 -g])`.

Ключевое следствие:

- `accel_body` должно соответствовать измерению акселерометра, а не
  произвольному линейному ускорению;
- в покое на земле официальный пример выдает не ноль, а вектор,
  соответствующий реакции опоры и силе тяжести.

## Отличия нашего прежнего цикла от официального

До TASK-25 в нашем контуре были обнаружены следующие расхождения с
официальным примером:

- MATLAB backend мог жить на собственном дискретном таймере
  `update_rate_hz`, а не только на `frame_rate`;
- JSON мог формироваться в длинном цикле как периодическая передача, а не
  строго как ответ на новый валидный кадр;
- использовались дополнительные служебные поля `no_time_sync` и
  `no_lockstep`, которых нет в официальном примере;
- в качестве основного поля ориентации использовался `quaternion`, тогда
  как официальный MATLAB-пример использует `attitude`;
- логика `accel_body` проверялась диагностическими вариантами, но не была
  явно сведена к формуле `SIM_multicopter.m`;
- семантика `frame_count` была размазана по верхнему сценарию, а не
  оформлена как отдельный lockstep-транспортный слой.

## Правила, переносимые в TASK-25

В TASK-25 в основной путь интегрируются следующие правила официального
примера:

- MATLAB слушает порт `9002`;
- один валидный бинарный кадр вызывает ровно один шаг физики;
- на один шаг физики формируется ровно один JSON-ответ;
- шаг интегрирования задается `frame_rate` с ограничением `max_timestep`;
- `frame_count` используется для учета дублей, пропусков и перезапуска;
- JSON формируется в минимальном официальном составе полей;
- `accel_body` приводится к формуле официального `SIM_multicopter.m`.
