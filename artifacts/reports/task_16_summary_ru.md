# TASK-16. Проверка управляемого режима ArduPilot на математической модели БВС

## Назначение задачи

TASK-16 продолжает работы после TASK-15, в котором был подтвержден
устойчивый 20-секундный обмен пакетами `ArduPilot SITL <-> MATLAB`.

Цель TASK-16 состоит не в подтверждении устойчивого автоматического
полета, а в проверке следующей инженерной цепочки:

`ArduPilot SITL -> длительности импульсов ШИМ -> команды частоты
вращения винтов -> математическая модель движения -> JSON/UDP ->
ArduPilot SITL`.

На данном этапе необходимо было:

- проверить запуск стенда в рабочем состоянии;
- зафиксировать состояние `Mission Planner`;
- проверить наличие `MAVLink HEARTBEAT`, если это возможно без помех
  основному потоку;
- записать историю выходов ШИМ `ArduPilot`;
- попытаться выполнить взведение;
- проверить, изменяются ли команды ШИМ после взведения;
- оценить возможность подтверждения порядка двигателей;
- выполнить первый управляемый расчетный прогон только при наличии
  подтвержденного взведения.

## Результат автоматической попытки взведения

Автоматическая попытка взведения выполнялась через `pymavlink` внутри
`WSL` при непрерывном JSON-обмене с математической моделью.

Зафиксирован следующий отказ:

- `HEARTBEAT` по `TCP 5762` получен;
- `ack_result = 4`;
- `arm_succeeded = false`;
- последнее сообщение причины:
  `Arm: Main loop slow (42Hz < 50Hz)`.

Также в журнале `ArduPilot` получено сообщение:

- `Arm: Gyro 0 rate 65Hz < loop rate*1.8 90Hz`

Эти два сообщения являются первой установленной технической причиной,
почему TASK-16 не дошел до управляемого режима:

- фактическая частота JSON-обмена математической модели недостаточна для
  прохождения предвзлетной проверки частоты гироскопа;
- даже после понижения `SCHED_LOOP_RATE` до `50 Гц` `ArduPilot` видит
  недостаточную частоту и сообщает о замедлении главного цикла.

Это решение является намеренно консервативным:

  `ArduPilot -> ШИМ -> силы/моменты -> изменение состояния модели` как
  подтвержденную.

## Следующий этап

Следующий шаг должен быть направлен именно на устранение причины отказа
взведения, а не на формальное расширение архитектуры.

Приоритетные действия:

1. измерить фактическую частоту отправки JSON-кадров со стороны
   `MATLAB` с более высокой целевой частотой обновления;
2. согласовать допустимую частоту главного цикла `ArduPilot SITL` для
   текущей математической модели;
3. при необходимости выполнить операторскую проверку причин отказа
   взведения в `Mission Planner`;
4. только после подтвержденного взведения повторить анализ порядка
   двигателей и первый управляемый расчетный прогон.
