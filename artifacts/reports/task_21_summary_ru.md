# TASK-21. Устранение отказа `Arm: 3D Accel calibration needed`

## 1. Что сделано

В рамках TASK-21 выполнены:

- диагностика фактических параметров калибровки акселерометра после
  загрузки профиля `tools/ardupilot/wsl/task20_single_accel_enable_mask.parm`;
- сравнение этих параметров со штатным внутренним `ArduPilot SITL`,
  который способен к взведению;
- штатная простая калибровка акселерометра в JSON-режиме через
  `MAVLink`;
- повторная проверка взведения после калибровки;
- построение графиков по параметрам калибровки и по результату
  повторной команды `arm`.

## 2. Исходная проблема после TASK-20

После TASK-20 было подтверждено:

- устойчивый обмен `JSON/UDP` в режиме `ArduPilot + MATLAB-модель`
  сохранен;
- отказ `Arm: Accels inconsistent` снят профилем
  `INS_ENABLE_MASK = 1`, `INS_USE = 1`, `INS_USE2 = 0`, `INS_USE3 = 0`;
- новой причиной отказа стало сообщение
  `Arm: 3D Accel calibration needed`;
- `COMMAND_ACK = 4`;
- `motor_pwm_us = 1000..1000 us`;
- `motor_cmd_radps = 0..0 рад/с`.

## 3. Что показала диагностика параметров

Сценарий `diagnose_ardupilot_accel_calibration_params.m` показал, что в
JSON-режиме после загрузки профиля TASK-20 фактически присутствуют:

- `INS_ENABLE_MASK = 1`;
- `INS_USE = 1`;
- `INS_USE2 = 0`;
- `INS_USE3 = 0`;
- `INS_ACC_ID = 2753028`;
- `INS_ACCOFFS_X/Y/Z = 0.001`;
- `INS_ACCSCAL_X/Y/Z = 1.001`.

Сценарий `compare_internal_sitl_accel_calibration_params.m` показал, что
в штатном внутреннем `SITL`:

- взведение подтверждено;
- `AHRS_ORIENTATION = 0`;
- `INS_ENABLE_MASK = 127`;
- `INS_USE2 = 1`;
- `INS_ACC2_ID = 2753036`.

Ключевой вывод сравнения:

- числовые коэффициенты калибровки основного акселерометра в JSON-режиме
  и в штатном `SITL` практически совпадают;
- однако JSON-режим с профилем TASK-20 использует только один
  акселерометр, тогда как штатный `SITL` использует более широкую
  конфигурацию ИНС;
- само наличие коэффициентов не снимало сообщение
  `3D Accel calibration needed`.

## 4. Штатная простая калибровка

Проверены попытки:

1. `MAVProxy accelcalsimple`;
2. `MAVProxy ground`;
3. `MAVLink MAV_CMD_PREFLIGHT_CALIBRATION` для простой калибровки.

Фактически выбран путь:

- метод: `mavlink_simple`;
- `COMMAND_ACK = 0`;
- устойчивый обмен во время калибровки не потерян.

Параметрический профиль
`tools/ardupilot/wsl/task21_single_accel_calibrated.parm`
не применялся, поскольку штатная простая калибровка выполнилась.

## 5. Повторная проверка взведения после калибровки

После штатной простой калибровки выполнен повторный практический прогон.

Фактические результаты:

- `valid_rx_count = 502`;
- `response_tx_count = 501`;
- `last_frame_count = 989`;
- сообщение `Arm: 3D Accel calibration needed` больше не наблюдалось;
- `arm = нет`;
- `COMMAND_ACK = 4`;
- первая причина отказа:
  `Arm: Gyro 0 rate 478Hz < loop rate*1.8 720Hz`.

Диапазоны выходных команд:

- `motor_pwm_us = 1000..1000 us`;
- `motor_cmd_radps = 0..0 рад/с`.

Следовательно:

- отказ по обязательной калибровке акселерометра устранен;
- ненулевой ШИМ и ненулевые команды частоты вращения винтов не получены;
- задача полного прохождения `arm` в TASK-21 не завершена из-за нового
  блокера по частоте гироскопа.

## 7. Что не является летной валидацией

Полученные результаты не являются:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением работоспособности замкнутого контура управления по
  внешней модели.

Подтвержден только следующий инженерный результат:

- штатная простая калибровка акселерометра в режиме
  `ArduPilot JSON + MATLAB-модель` выполнима;
- именно она снимает сообщение
  `Arm: 3D Accel calibration needed`;
- следующий блокирующий фактор относится к частоте гироскопа, а не к
  калибровке акселерометра.

## 8. Следующий этап

Следующий этап должен быть посвящен отдельной диагностике отказа:

- `Arm: Gyro 0 rate ...`;
- сопоставлению частоты выдачи инерциальных данных со значением
  `SCHED_LOOP_RATE`;
- выбору минимального исправления без отключения `ARMING_CHECK`.
