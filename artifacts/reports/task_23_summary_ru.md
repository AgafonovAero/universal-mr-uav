# TASK-23. Первый взлетный прогон ArduPilot JSON на модели БВС

## 1. Что сделано

В рамках TASK-23 выполнены:

- диагностика фактических параметров моторных выходов ArduPilot в режиме `ArduPilot JSON + MATLAB-модель`;
- расчет преобразования `ШИМ -> команда частоты вращения -> тяга -> отношение тяги к весу`;
- опыт `arm` без команды взлета;
- опыт `GUIDED takeoff 1 м`;
- дополнительный диагностический опыт `throttle step`, поскольку автоматическая команда `takeoff`
  не дала роста ШИМ;
- подготовка диагностического профиля тяги `task23_hover_thrust_diagnostic.parm`;
- формирование исходных журналов, таблиц и графиков TASK-23.

## 2. Исходное состояние после TASK-22

После TASK-22 в режиме `ArduPilot JSON + MATLAB-модель` было подтверждено:

- устойчивый `JSON/UDP`-обмен;
- принятое взведение:
  `arm = да`,
  `COMMAND_ACK = 0`;
- диапазон `motor_pwm_us = 1000..1100 мкс`;
- диапазон `motor_cmd_radps = 0..90 рад/с`;
- предварительное соответствие каналов двигателям `1..4`.

Задача TASK-23 состояла уже не в снятии очередного предполетного блокера ArduPilot,
а в проверке причинно-следственной цепи:

`команда ArduPilot -> ШИМ -> частота вращения винтов -> тяга -> изменение высоты модели`.

## 4. Расчет ШИМ -> частота вращения -> тяга

Сценарий `scripts/diagnose_pwm_to_thrust_balance.m` построил тяговый баланс по текущему
преобразованию `uav.ardupilot.pwm_to_motor_radps` и текущим коэффициентам винтомоторной группы.

Ключевые результаты:

- расчетный уровень зависания модели: около `1600 мкс`;
- в точке `1600 мкс` отношение суммарной тяги к весу равно `0.9512`;
- при `1650 мкс` отношение уже превышает единицу и составляет `1.1163`;
- эквивалент текущего `MOT_THST_HOVER = 0.39` по линейной шкале ArduPilot: `1390 мкс`;
- при `1100 мкс` отношение суммарной тяги к весу равно только `0.0264`;
- при `2000 мкс` отношение достигает `2.6422`.

Вывод:

- сама математическая модель БВС допускает отрыв по тяге, поскольку при больших командах ШИМ
  запас тяги существенно превышает вес;
- текущий диапазон `1000..1100 мкс`, наблюдавшийся после `arm`, принципиально недостаточен
  для отрыва;
- основное инженерное расхождение состоит не в том, что модель вообще не может взлететь,
  а в том, что ArduPilot в диагностированных опытах не вышел на уровень ШИМ, близкий к
  расчетному зависанию;
- текущее значение `MOT_THST_HOVER = 0.39` эквивалентно `1390 мкс` и заметно ниже
  расчетного уровня зависания модели.

## 5. Результат idle arm

Сценарий `scripts/run_ardupilot_json_arm_idle_observation.m` выполнил устойчивый прогон
без команды взлета.

Получено:

- `arm = да`;
- `COMMAND_ACK = 0`;
- `valid_rx_count = 762`;
- `response_tx_count = 761`;
- `last_frame_count = 1261`;
- диапазон `motor_pwm_us = 1000..1100 мкс`;
- диапазон `motor_cmd_radps = 0..90 рад/с`;
- максимальное отношение суммарной тяги к весу: `0.026422`.

Это подтверждает первую часть причинно-следственной цепи:

- ArduPilot после принятой команды `arm` действительно выдает ненулевой ШИМ;
- преобразование ШИМ в команды частоты вращения работает;
- математическая модель получает ненулевые команды на винтомоторную группу.

Однако для вертикального отрыва этого уровня тяги недостаточно.
Рост высоты вверх не подтвержден; напротив, по текущему определению высоты модель уходит вниз,
что согласуется с недостаточной суммарной тягой.

## 6. Результат GUIDED takeoff 1 м

Сценарий `scripts/run_ardupilot_json_guided_takeoff_1m.m` выполнил попытку:

- перехода в режим `GUIDED`;
- взведения;
- команды `takeoff 1 м`.

Фактический результат:

- режим `GUIDED` наблюдался в телеметрии: да;
- `valid_rx_count = 765`;
- `response_tx_count = 765`;
- `last_frame_count = 1264`;
- `arm = нет`;
- `COMMAND_ACK` для `arm = 4`;
- `takeoff COMMAND_ACK = NaN`;
- первая причина отказа:
  `Arm: GPS 1: Bad fix`;
- диапазон `motor_pwm_us = 1000..1000 мкс`;
- диапазон `motor_cmd_radps = 0..0 рад/с`;
- максимальное отношение суммарной тяги к весу: `0.0000`;
- рост высоты вверх: нет.

Следовательно, команда `GUIDED takeoff 1 м` на текущем этапе не стала источником роста тяги,
поскольку блокировка произошла еще на этапе `arm` по причине отсутствия приемлемой GPS-фиксации.

## 7. Результат throttle step

Поскольку `GUIDED takeoff` не дал роста ШИМ, выполнен дополнительный сценарий
`scripts/run_ardupilot_json_throttle_step_response.m`.

Фактический результат последнего актуального прогона:

- `valid_rx_count = 855`;
- `response_tx_count = 855`;
- `last_frame_count = 1355`;
- `arm = нет`;
- `COMMAND_ACK = 4`;
- первая причина отказа:
  `Arm: Gyro 0 rate 348Hz < loop rate*1.8 360Hz`;
- диапазон `motor_pwm_us = 1000..1000 мкс`;
- диапазон `motor_cmd_radps = 0..0 рад/с`;
- рост тяги не подтвержден.

Следовательно, текущий автоматический путь диагностического ступенчатого воздействия по газу
не дал прироста ШИМ, поскольку на этом опыте снова проявился предполетный блокер по частоте
гироскопа.

## 8. Диагностический профиль тяги

Для проверки гипотезы о недостаточном уровне тяги был подготовлен диагностический профиль:

- `tools/ardupilot/wsl/task23_hover_thrust_diagnostic.parm`

с изменениями:

- `MOT_THST_HOVER = 0.60`;
- `MOT_SPIN_ARM = 0.55`;
- `MOT_SPIN_MIN = 0.60`.

Этот профиль не выбран как основной, поскольку в диагностическом прогоне он не улучшил
результат и ухудшил прохождение команды `arm`. Поэтому он сохраняется только как
неудачный диагностический вариант, а не как штатная настройка.

## 10. Итог TASK-23

По итогам TASK-23 подтверждено:

- устойчивый обмен `ArduPilot JSON + MATLAB-модель` сохраняется;
- после `arm` ArduPilot действительно выдает ШИМ выше минимального уровня;
- цепочка `arm -> PWM -> motor_cmd_radps -> тяга` подтверждена;
- на уровне `1000..1100 мкс` тяги недостаточно для отрыва модели;
- автоматический `GUIDED takeoff 1 м` на текущем шаге не проходит из-за
  `Arm: GPS 1: Bad fix`;
- диагностический `throttle step` в последнем актуальном прогоне не проходит из-за
  повторного блокера по частоте гироскопа.

Разрешено сделать только следующий инженерный вывод:

- первичная реакция на команду взведения подтверждена;
- первый управляемый взлетный прогон выполнен как диагностическая попытка;
- рост высоты вверх по команде взлета пока не подтвержден.

## 11. Что не является летной валидацией

Результаты TASK-23 не являются:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением штатной пригодности контура управления высотой;
- подтверждением корректной работы режима `GUIDED` на реальном объекте.

## 12. Следующий этап

Следующий этап должен быть посвящен не расчету тягового баланса, а устранению
конкретных причин, из-за которых ArduPilot не выходит на рабочую тягу при команде взлета:

- диагностике `GPS 1: Bad fix` в режиме `GUIDED`;
- стабилизации предполетной проверки по частоте гироскопа в диагностическом канале
  ручной команды газа;
- после снятия этих блокеров — повторному опыту `GUIDED takeoff` с анализом роста тяги,
  высоты и углового движения без заявлений о летной валидации.
