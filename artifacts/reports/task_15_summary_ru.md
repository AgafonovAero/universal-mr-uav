# TASK-15. Подтверждение фактического обмена ArduPilot и MATLAB-модели

## Назначение задачи

TASK-15 предназначен для перехода от безопасных проверочных прогонов
TASK-14 к фактическому использованию уже созданного стенда на данной
машине.

Целью этапа являлось не создание новой архитектуры, а подтверждение
следующих фактов:

- `ArduPilot SITL` действительно запускается в режиме `JSON`;
- `MATLAB` действительно принимает двоичный пакет от `ArduPilot`;
- `MATLAB` действительно выполняет ответную передачу строки `JSON`
  после принятого пакета;
- существующий расчетный сценарий может выполнить ограниченный по
  времени обмен с математической моделью движения;
- `Mission Planner` может быть запущен в составе стенда, а порт
  `MAVLink UDP 14550` действительно используется.

На данном этапе не изменялись:

- математическая модель движения;
- алгоритмы оценивания состояния;
- коэффициенты;
- модели `Simulink`.

## Сетевой адрес Windows-хоста для WSL2

Для `WSL2` нельзя было использовать `127.0.0.1` без проверки, так как
обмен между Windows и `Ubuntu` идет через виртуальный сетевой интерфейс.

По журналу `artifacts/logs/task_15_detect_wsl_network.txt`
подтверждено:

- интерфейс Windows:
  `vEthernet (WSL (Hyper-V firewall))`;
- адрес Windows-хоста, видимый из `WSL2`:
  `172.19.208.1`;
- адрес `Ubuntu` внутри `WSL2`:
  `172.19.219.165`;
- шлюз по умолчанию внутри `Ubuntu`:
  `172.19.208.1`.

Именно поэтому для фактического запуска были приняты следующие
параметры:

- адрес `JSON`-модели:
  `172.19.208.1:9002`;
- вывод `MAVLink`:
  `udp:172.19.208.1:14550`.

Выбран именно этот адрес, потому что он одновременно:

- является адресом интерфейса `vEthernet (WSL)` в Windows;
- виден из `Ubuntu` как шлюз по умолчанию;
- обеспечивает фактический прием пакетов на стороне Windows.

## Команда запуска ArduPilot SITL

Фактический запуск стенда выполнялся через сценарий:

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\ardupilot\windows\Start-ArduPilotJsonSitl.ps1 -DistroName Ubuntu -Ip 172.19.208.1 -MavlinkPort 14550 -NoConsole -NoMap -Execute
```

По журналу
`artifacts/logs/task_15_start_ardupilot_json_sitl_real.txt`
подтверждено, что фактическая команда внутри `WSL` была сформирована
для следующего режима:

- `JSON:172.19.208.1`;
- `udpclient:172.19.208.1:14550`;
- `arducopter` запускался как процесс `SITL`.

Подтвержденный экземпляр процесса:

- `arducopter --model JSON:172.19.208.1 --speedup 1 --slave 0 --serial0=udpclient:172.19.208.1:14550 --defaults ... --sim-address=172.19.208.1 -I0`

Следовательно, факт запуска `ArduPilot SITL` подтвержден.
