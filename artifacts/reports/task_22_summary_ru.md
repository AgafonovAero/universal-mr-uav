# TASK-22. Устранение отказа взведения ArduPilot по частоте гироскопа

## 1. Что сделано

В рамках TASK-22 выполнены:

- диагностика фактической частоты гироскопа и текущего значения
  `SCHED_LOOP_RATE` в режиме `ArduPilot JSON + MATLAB-модель`;
- подготовка и проверка профилей `SCHED_LOOP_RATE = 250`, `200`, `100`;
- серия практических опытов по взведению на каждом профиле;
- дополнительная проверка реакции винтомоторной группы после принятия
  команды `arm`;
- анализ порядка двигателей;
- первый управляемый расчетный прогон без заявлений о летной валидации;
- построение графиков по частоте гироскопа и результатам опытов.

## 2. Исходная проблема после TASK-21

После снятия отказа `Arm: 3D Accel calibration needed` штатной простой
калибровкой акселерометра устойчивый `JSON/UDP`-обмен сохранялся, но
взведение блокировалось сообщением:

`Arm: Gyro 0 rate 430Hz < loop rate*1.8 720Hz`

При этом:

- `COMMAND_ACK = 4`;
- `motor_pwm_us = 1000..1000 us`;
- `motor_cmd_radps = 0..0 рад/с`.

## 3. Диагностика частоты гироскопа

Сценарий `scripts/diagnose_ardupilot_gyro_loop_rate.m` показал:

- `SCHED_LOOP_RATE = 400 Гц`;
- `SIM_RATE_HZ = 1200 Гц`;
- фактическая частота гироскопа по сообщению ArduPilot: `430 Гц`;
- требуемая минимальная частота: `1.8 * 400 = 720 Гц`;
- максимальное допустимое значение `SCHED_LOOP_RATE` при такой частоте:
  `238 Гц`;
- устойчивый обмен сохранен:
  `valid_rx_count = 632`,
  `response_tx_count = 631`,
  `last_frame_count = 1129`.

Следовательно, первопричина отказа заключалась в завышенном значении
главного цикла ArduPilot относительно фактической частоты поступления
гироскопических данных в JSON-режиме.

## 4. Какие значения SCHED_LOOP_RATE проверены

Проверены профили:

- `task20_single_accel_enable_mask.parm` с фактическим
  `SCHED_LOOP_RATE = 400`;
- `task22_sched_loop_250.parm`;
- `task22_sched_loop_200.parm`;
- `task22_sched_loop_100.parm`.

Сценарий `scripts/verify_ardupilot_sched_loop_profiles.m` подтвердил,
что значения `250`, `200` и `100` фактически применяются после загрузки.

## 6. Выбранный профиль

Для основного режима TASK-22 выбран профиль:

- `tools/ardupilot/wsl/task22_sched_loop_200.parm`

с параметром:

- `SCHED_LOOP_RATE = 200`.

Причина выбора:

- это наибольшее значение `SCHED_LOOP_RATE`, которое снимает блокер по
  частоте гироскопа и при этом дает принятое взведение;
- профиль `250 Гц` еще не проходит условие `gyro_rate >= 1.8 * loop_rate`;
- профиль `100 Гц` не требуется как основной, поскольку снижает частоту
  цикла сильнее, чем необходимо для прохождения предполетной проверки.

## 7. Проверка после arm

После выбора профиля `SCHED_LOOP_RATE = 200` дополнительно подтверждено:

- в проверке `run_ardupilot_arm_pwm_response.m`
  `COMMAND_ACK = 0`,
  диапазон `motor_pwm_us = 1000..1100 us`,
  диапазон `motor_cmd_radps = 0..90 рад/с`;
- в `analyze_ardupilot_motor_order.m` прямое соответствие каналов и
  двигателей `1..4` подтверждено;
- в `run_ardupilot_first_controlled_response.m`
  `valid_rx_count = 507`,
  `json_tx_count = 1731`,
  `response_tx_count = 506`,
  `last_frame_count = 1637`,
  состояние модели остается конечным.

## 9. Что не является летной валидацией

Полученный результат не является:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением штатной эксплуатации замкнутого контура управления.

Подтвержден только следующий инженерный результат:

- блокер `Gyro 0 rate ... < loop rate*1.8 ...` снят параметрическим
  уменьшением `SCHED_LOOP_RATE` до `200`;
- команда `arm` принята;
- в режиме `ArduPilot JSON + MATLAB-модель` появились ненулевой ШИМ и
  ненулевые команды частоты вращения винтов.

## 10. Следующий этап

Следующий этап должен быть посвящен уже не предполетной проверке
ArduPilot, а проверке поведения объекта управления после arm:

- устойчивости ненулевого ШИМ в более длинном окне наблюдения;
- связи между ШИМ, частотой вращения винтов и изменением состояния
  математической модели движения;
- дальнейшей отработке режимов управления без заявлений о летной
  валидации.
