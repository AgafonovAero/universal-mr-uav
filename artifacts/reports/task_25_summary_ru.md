# TASK-25. Приведение ArduPilot JSON к официальной MATLAB-схеме

## Что исправлено относительно предыдущего состояния PR №25

В рамках уточнения TASK-25 основной контур `ArduPilot JSON SITL <-> MATLAB`
сохранен в официальной lockstep-схеме:

- один валидный двоичный пакет ArduPilot вызывает ровно один шаг физики;
- один шаг физики вызывает ровно один JSON-ответ;
- MATLAB слушает `UDP 9002`;
- JSON не отправляется без нового валидного входного пакета;
- `frame_rate` используется для расчета шага интегрирования;
- `frame_count` используется для учета дублей, пропусков и сброса цикла.

Критическое расхождение по `imu.accel_body` в состоянии покоя на земле
устранено. Ранее отчет ошибочно показывал для `ground_rest` значение
`current = [0 0 0]`, хотя официальный режим должен выдавать измерение
акселерометра, совпадающее с эталонным примером ArduPilot.

## Результат приведения accel_body к официальному примеру

Повторное сравнение с официальным MATLAB-примером ArduPilot показало:

- `ground_rest`: `current = [0 0 -9.81]`, `official = [0 0 -9.81]`,
  `delta_norm = 0`;
- `hover`: расхождение равно нулю;
- `vertical_accel_up`: расхождение равно нулю;
- `vertical_accel_down`: расхождение равно нулю;
- `small_roll_pitch_hover`: расхождение равно нулю.

Итог: в основном официальном режиме `accel_body` приведен к эталонной
формуле ArduPilot для всех пяти проверочных состояний, включая покой на
земле.

Отдельный диагностический режим наземного зануления сохранен только как
вспомогательный и не используется в штатных сценариях TASK-25.

## Результат 20-секундного официального lockstep-прогона

Повторный 20-секундный прогон в официальной схеме дал:

- `valid_rx_count = 3188`;
- `json_response_tx_count = 3187`;
- `missed_frame_count = 0`;
- `duplicate_frame_count = 0`;
- `invalid_rx_count = 0`;
- `last_frame_count = 3187`;
- `max_altitude_m = 0`.

Полученные счетчики подтверждают, что контур работает именно в режиме
`packet -> step -> response`, без самопроизвольной периодической отправки
JSON.

## Результат попыток взведения после 20, 60 и 120 секунд

Сценарий взведения был повторен при непрерывном официальном lockstep-обмене
с тремя моментами попытки `arm`:

- после `20` секунд обмена;
- после `60` секунд обмена;
- после `120` секунд обмена.

Результат:

- `arm_after_20_s = нет`, `COMMAND_ACK = 4`,
  первая причина: `Arm: System not initialised`;
- `arm_after_60_s = нет`, `COMMAND_ACK = 4`,
  первая причина: `Arm: System not initialised`;
- `arm_after_120_s = нет`, `COMMAND_ACK = 0`,
  в `STATUSTEXT` наблюдалось сообщение `Arming motors`, однако устойчивое
  состояние `armed = true` не зафиксировано и сценарий не подтверждает
  фактическое завершенное взведение.

Дополнительные измерения по этому прогону:

- `valid_rx_count = 5855`;
- `json_response_tx_count = 5854`;
- `missed_frame_count = 0`;
- `duplicate_frame_count = 0`;
- `last_frame_count = 5854`;
- `motor_pwm_us = 0..1058 мкс`;
- `motor_cmd_radps = 0..52.2 рад/с`.

Практический вывод: после исправления `accel_body` первый устойчиво
наблюдаемый блокер остается тем же:

`Arm: System not initialised`

Следовательно, это уже следующий реальный блокер ArduPilot после устранения
расхождения по наземному `accel_body`.

## A/B-проверка влияния diagnostic ground clamp

Для проверки гипотезы выполнено сравнение двух вариантов:

- вариант `A`: диагностический `ground_clamp = true`;
- вариант `B`: официальный режим `ground_clamp = false`.

В обоих вариантах получено:

- `arm = нет`;
- `COMMAND_ACK = 4`;
- первая причина: `Arm: System not initialised`.

Итог A/B-проверки: диагностическое наземное зануление не является причиной
текущего отказа `System not initialised`.

## Что не является летной валидацией

Полученный результат не является:

- летной валидацией;
- подтверждением автоматического взлета;
- подтверждением устойчивого автоматического полета;
- подтверждением штатного прохождения полного цикла `arm -> takeoff`.

Подтверждено только:

- корректное приведение `accel_body` к официальному MATLAB-примеру
  ArduPilot;
- работоспособность официального lockstep-контура `binary packet -> one
  physics step -> one JSON response`;
- наличие следующего инженерного блокера
  `Arm: System not initialised`.

## Следующий этап

Следующий этап должен быть направлен уже не на исправление `accel_body`, а
на диагностику причин незавершенной инициализации ArduPilot после запуска
официального MATLAB backend:

- выяснить, какие внутренние подсистемы остаются в состоянии
  `not initialised`;
- проверить статус EKF/AHRS и сопутствующих предполетных проверок;
- повторить попытку `arm` только после явного подтверждения завершенной
  инициализации.
