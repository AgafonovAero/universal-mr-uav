# Проверка управляемого режима ArduPilot на математической модели БВС

## Назначение TASK-16

TASK-16 переводит проект от проверки устойчивого обмена пакетами к
проверке причинно-следственной цепочки управления:

`Mission Planner -> ArduPilot SITL -> команды ШИМ -> MATLAB-модель ->
JSON/UDP -> ArduPilot SITL`.

На данном этапе проверяется не летная пригодность и не устойчивый
автоматический полет, а сам факт того, что внешний программный комплекс
управления способен:

- выдавать длительности импульсов ШИМ;
- передавать их в математическую модель движения;
- инициировать изменение команд частоты вращения винтов;
- сохранять работоспособность расчетного контура при длительном обмене.

## Отличие TASK-16 от TASK-15

TASK-15 подтвердил:

- установку `Ubuntu` в `WSL`;
- установку `ArduPilot` во внешнем каталоге `/home/oaleg/src/ardupilot`;
- запуск `ArduPilot SITL` в режиме `JSON`;
- устойчивый 20-секундный обмен `ArduPilot SITL <-> MATLAB` без
  аварийного завершения процесса `arducopter`;
- запуск `Mission Planner`.

TASK-16 добавляет к этому новую инженерную проверку:

- наблюдение выходов ШИМ `ArduPilot`;
- попытку взведения с сохранением фактической причины отказа;
- проверку преобразования ШИМ в команды частоты вращения винтов;
- анализ достаточности данных для проверки порядка двигателей;
- попытку первого управляемого расчетного прогона.

## Схема стенда

В ходе TASK-16 использовалась следующая схема взаимодействия:

1. `ArduPilot SITL` запускается внутри `WSL`.
2. `ArduPilot SITL` направляет двоичный пакет выходов на порт `9002`
   Windows-хоста `172.19.208.1`.
3. `MATLAB` принимает пакет, преобразует длительности импульсов ШИМ в
   команды частоты вращения винтов и выполняет шаг математической модели
   движения.
4. `MATLAB` формирует строку `JSON` с данными подсистемы датчиков,
   кинематического состояния и ориентации.
5. После принятого двоичного пакета строка `JSON` отправляется обратно в
   `ArduPilot SITL`.
6. `Mission Planner` получает поток `MAVLink` по адресу
   `udp:172.19.208.1:14550`.

## Фактическая команда запуска ArduPilot SITL

Для TASK-16 использовалась следующая команда:

```bash
cd /home/oaleg/src/ardupilot/ArduCopter
../build/sitl/bin/arducopter --model JSON:172.19.208.1 --speedup 1 --slave 0 --serial0=udpclient:172.19.208.1:14550 --defaults ../Tools/autotest/default_params/copter.parm,/mnt/d/WORK/AI_SOLUTION/universal-mr-uav-task16/tools/ardupilot/wsl/task16_arducopter_loop_rate.parm --sim-address=172.19.208.1 -I0
```

Дополнительный файл параметров
`tools/ardupilot/wsl/task16_arducopter_loop_rate.parm` содержит только
одно переопределение:

```text
SCHED_LOOP_RATE,50
```

Это переопределение введено не для отключения защитных проверок, а для
согласования частоты главного цикла `ArduPilot SITL` с наблюдаемой
частотой JSON-обмена математической модели.

## Что проверяется на этапе управляемого режима

TASK-16 проверяет следующие факты:

- процесс `arducopter` остается жив во время расчетного обмена;
- `MATLAB` принимает больше 50 валидных двоичных пакетов;
- счетчик `frame_count` возрастает;
- каналы `motor_pwm_us` имеют размерность `4 x 1`;
- команды `motor_cmd_radps` конечны и формируются из `motor_pwm_us`;
- наземная станция управления запущена;
- при наличии данных анализируется достаточность возбуждения каналов для
  проверки порядка двигателей.

TASK-16 не проверяет:

- устойчивый автоматический полет;
- валидацию параметров контура управления;
- соответствие модели реальному летательному аппарату;
- летную годность.

## Проверка MAVLink HEARTBEAT

Для дополнительного программного контроля создан сценарий
`scripts/check_mavlink_heartbeat.m`.

Он прослушивает отдельный UDP-порт `14552` и ищет сообщение `HEARTBEAT`
протокола `MAVLink` версии 1 или 2 без разворачивания полного стека
`MAVLink`.

На текущем запуске получен следующий результат:

- `Mission Planner` запущен;
- локально занят UDP-порт `14551` процессом `MissionPlanner.exe`;
- на дополнительном порту `14552` программное подтверждение `HEARTBEAT`
  не получено.

Следовательно, на данном этапе подтверждается запуск процесса наземной
станции управления, но не подтверждается программное обнаружение
`HEARTBEAT` на дополнительном порту.

## Наблюдение ШИМ и команд винтомоторной группы

Сценарий `scripts/run_ardupilot_pwm_observation.m` выполняет
20-секундный прогон обмена и сохраняет:

- время `time_s`;
- счетчик `frame_count`;
- матрицу `pwm_us`;
- матрицу `motor_pwm_us`;
- матрицу `motor_cmd_radps`;
- историю состояния модели;
- историю оценивания состояния;
- статус обмена.

По текущему прогону подтверждено:

- валидные двоичные пакеты приходят устойчиво;
- `frame_count` возрастает;
- все четыре канала `motor_pwm_us` присутствуют;
- значения `motor_pwm_us` остаются на минимальном уровне `1000 us`;
- команды `motor_cmd_radps` конечны, но равны нулю.

Это означает, что обмен работает устойчиво, но `ArduPilot` не перешел в
состояние, при котором выдаются ненулевые команды частоты вращения
винтов.

## Попытка взведения

Для автоматической попытки взведения создан сценарий
`scripts/try_ardupilot_arm.m`.

Он использует `pymavlink` внутри `WSL` и соединяется с `ArduPilot SITL`
по локальному TCP-порту `5762`.

На наиболее содержательной попытке, выполненной при непрерывном
JSON-обмене, зафиксирован отказ `ArduPilot` со следующими сообщениями:

- `Arm: Gyro 0 rate 65Hz < loop rate*1.8 90Hz`
- `Arm: Main loop slow (42Hz < 50Hz)`

Тем самым TASK-16 установил первую техническую причину, мешающую
переходу к управляемому режиму: текущая частота обмена со стороны
математической модели недостаточна для прохождения предвзлетных проверок
`ArduPilot`.

## Проверка порядка двигателей

Сценарий `scripts/analyze_ardupilot_motor_order.m` использует историю
`motor_pwm_us` и `motor_cmd_radps` и сравнивает ее с геометрией `quad-X`
из параметров модели.

На текущем прогоне автоматического подтверждения порядка двигателей нет,
потому что:

- диапазоны ШИМ по всем четырем каналам равны нулю;
- диапазоны команд частоты вращения также равны нулю;
- возбуждение каналов недостаточно для анализа знака по рысканию и
  перестановки соответствий.

По этой причине TASK-16 честно фиксирует:

- порядок двигателей не подтвержден;
- автоматическая перестановка каналов не выполнялась.

## Критерий первого управляемого расчетного прогона

Для первого управляемого прогона нужен одновременно следующий набор
фактов:

- `ArduPilot SITL` запущен;
- автоматическое или операторское взведение подтверждено;
- `motor_pwm_us` отличается от минимального фиксированного уровня;
- `motor_cmd_radps` не нулевые и конечны;
- состояние модели изменяется конечным образом без `NaN` и `Inf`.

На текущем состоянии TASK-16 этот критерий не выполнен, поскольку
взведение не подтверждено.

## Что не является летной валидацией

Даже после выполнения TASK-16 нельзя утверждать:

- устойчивый автоматический полет;
- настройку контуров управления;
- соответствие модели конкретному реальному БВС;
- летную валидацию.

TASK-16 относится только к проверке управляемого режима в контуре
программного моделирования.

## Порядок запуска

Для воспроизведения стенда на текущем этапе следует:

1. запустить `ArduPilot SITL` с адресом `JSON:172.19.208.1`;
2. запустить `Mission Planner`;
3. выполнить `scripts/run_ardupilot_pwm_observation.m`;
4. выполнить `scripts/try_ardupilot_arm.m`;
5. при необходимости проверить `scripts/check_mavlink_heartbeat.m`;
6. затем выполнить анализ истории ШИМ и построение графиков.

Если автоматическое взведение снова не проходит, оператору необходимо в
`Mission Planner`:

1. подключиться по `UDP`;
2. выбрать порт `14550`;
3. дождаться связи;
4. открыть сообщения о причине отказа взведения;
5. только после этого принимать решение о дальнейшем снижении частоты
   главного цикла, повышении частоты JSON-обмена или настройке условий
   арминга.

## Ограничения текущего этапа

На момент завершения TASK-16 остаются следующие ограничения:

- `HEARTBEAT` на дополнительном порту программно не подтвержден;
- автоматическое взведение не выполнено;
- `motor_pwm_us` остаются на минимальном уровне;
- порядок двигателей не подтвержден;
- первый управляемый расчетный прогон не выполнен.

## Следующий этап

Следующим инженерным шагом должен стать не переход к летной валидации, а
устранение причины отказа взведения:

- либо повышение фактической частоты JSON-обмена со стороны
  MATLAB-модели;
- либо дальнейшее согласование параметров `ArduPilot SITL`, связанных с
  частотой главного цикла и условиями предвзлетных проверок;
- либо операторское подтверждение допустимого режима взведения в
  `Mission Planner` с сохранением журналов.
