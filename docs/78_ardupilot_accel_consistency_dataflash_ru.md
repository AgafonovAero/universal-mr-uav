# TASK-20. Диагностика согласованности акселерометров ArduPilot по DataFlash

## Назначение

TASK-20 продолжает практический стенд `ArduPilot SITL + MATLAB-модель universal-mr-uav + Mission Planner`
после TASK-19.

На входе TASK-20 были подтверждены:

- устойчивый обмен `JSON/UDP`;
- `valid_rx_count > 50`;
- `response_tx_count > 50`;
- воспроизводимый отказ взведения `Arm: Accels inconsistent`;
- отсутствие результата от простой замены соглашения `imu.accel_body`;
- отсутствие результата от задержки перед `arm`;
- отсутствие результата от профиля, где менялись только `INS_USE2/INS_USE3`.

Назначение TASK-20:

- получить фактический `DataFlash`-журнал `ArduPilot` в JSON-режиме;
- извлечь внутренние данные нескольких экземпляров ИНС и акселерометров;
- проверить, применяются ли параметры `INS_USE2`, `INS_USE3` и `INS_ENABLE_MASK`
  после загрузки;
- сравнить внутренние значения `ArduPilot` с ранее сохраненными `JSON`-кадрами MATLAB-модели;
- выбрать минимальное исправление только по фактическим данным журнала.

## Что означает отказ `Arm: Accels inconsistent`

Сообщение `Arm: Accels inconsistent` относится к внутренней проверке
согласованности акселерометров `ArduPilot`.

Для практического стенда `ArduPilot JSON + MATLAB-модель` это означает, что
подсистема инерциальных датчиков считает активные экземпляры акселерометров
несогласованными и не допускает взведение.

TASK-20 специально не лечит эту проблему отключением `ARMING_CHECK = 0`,
а ищет фактический источник рассогласования по внутреннему журналу `ArduPilot`.

Сохраненные результаты:

- `artifacts/reports/task_20_accel_instances.csv`
- `artifacts/reports/task_20_accel_instances.mat`

По фактическому `DataFlash`-журналу baseline-прогона подтверждено:

- активны экземпляры ИНС `IMU0` и `IMU1`;
- `INS_USE = 1`;
- `INS_USE2 = 1`;
- `INS_USE3 = 1`;
- `INS_ENABLE_MASK = 127`;
- максимальное различие между акселерометрами за весь интервал:
  `0.005905`;
- максимальное различие в окне перед первой попыткой взведения:
  `0.005905`.

Последние сообщения `MSG` в `DataFlash` включали:

- `EKF3 IMU0 initialised`;
- `EKF3 IMU1 initialised`;
- `Arm: Accels inconsistent`;
- `PreArm: Gyros inconsistent`.

Практический вывод:

- в baseline-режиме реально активны как минимум два экземпляра ИНС;
- исходный профиль TASK-19 менял только флаги использования, но не
  отключал сами внутренние экземпляры через маску включения.
- диагностически проблемным экземпляром признан второй активный
  акселерометр `IMU1`, поскольку только после его фактического исключения
  из активной маски исходное сообщение `Arm: Accels inconsistent` исчезло.

## Проверка фактического применения параметров ИНС

Сценарий:

- `scripts/verify_ardupilot_ins_params_after_boot.m`

Сохраненные результаты:

- `artifacts/reports/task_20_ins_params_after_boot.csv`
- `artifacts/logs/task_20_verify_ins_params_after_boot.txt`

Проверены три профиля:

1. `default`
2. `task19_single_imu`
3. `task20_enable_mask`

Результат:

- профиль `task19_single_imu` действительно устанавливал
  `INS_USE2 = 0`, `INS_USE3 = 0`;
- при этом `INS_ENABLE_MASK` после загрузки оставался равным `127`;
- следовательно, в предыдущей диагностике второй активный экземпляр ИНС
  не был фактически исключен;
- профиль `task20_enable_mask` после загрузки давал:
  `INS_USE = 1`, `INS_USE2 = 0`, `INS_USE3 = 0`, `INS_ENABLE_MASK = 1`.

Практический вывод:

- параметрический профиль одного ИНС из TASK-19 применялся не полностью;
- минимальным фактическим исправлением стало не просто изменение
  `INS_USE2/INS_USE3`, а реальное ограничение маски включения ИНС через
  `INS_ENABLE_MASK = 1`.

## Сравнение JSON-ускорений и внутренних значений ArduPilot

Сценарий:

- `scripts/compare_json_accel_with_ardupilot_log.m`

Сравнивались:

- `imu.accel_body` из `artifacts/reports/task_19_first_json_frames.jsonl`;
- внутренние значения `IMU0`;
- внутренние значения `IMU1`.

Сохраненные результаты:

- `artifacts/reports/task_20_json_vs_ardupilot_accel.csv`
- `artifacts/reports/task_20_json_vs_ardupilot_accel.mat`

Фактический результат:

- в первых 200 `JSON`-кадрах MATLAB-модель передает `imu.accel_body = [0, 0, 0]`;
- внутренние акселерометры `ArduPilot` в состоянии покоя тоже имеют значения,
  близкие к нулю;
- максимальное различие между `JSON accel_body` и внутренними значениями
  экземпляров ИНС составляет `0.004816`;
- максимальное различие норм между экземплярами ИНС составляет `0.002645`;
- различие больше `1 м/с²` не подтверждено.

Практический вывод:

- проблема не сводится к простому знаку `accel_body` или к грубой ошибке масштаба;
- по журналу различие численных значений мало;
- исходный отказ был связан именно с составом активных экземпляров ИНС и их
  внутренней конфигурацией, а не с грубым искажением `JSON`-ускорения.

## Выбранное исправление

В TASK-20 выбран диагностический параметрический профиль:

- `tools/ardupilot/wsl/task20_single_accel_enable_mask.parm`

Профиль задает:

- `INS_USE = 1`
- `INS_USE2 = 0`
- `INS_USE3 = 0`
- `INS_ENABLE_MASK = 1`
- `LOG_DISARMED = 1`

Причина выбора:

- только этот профиль фактически применился после загрузки так, чтобы
  `INS_ENABLE_MASK` стал равен `1`;
- после его применения сообщение `Arm: Accels inconsistent` исчезло;
- новая причина отказа стала другой:
  `Arm: 3D Accel calibration needed`.

Следовательно, TASK-20 устранил именно исходное рассогласование акселерометров
как первичный блокер, но не довел стенд до принятого взведения.
