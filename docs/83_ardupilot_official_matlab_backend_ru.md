# TASK-25. Приведение ArduPilot JSON к официальной MATLAB-схеме

## Цель

Цель TASK-25 состояла в переходе от самотактируемого контура обмена к
официальной схеме MATLAB backend ArduPilot:

`ArduPilot SITL -> двоичный пакет ШИМ -> один шаг физики MATLAB -> один JSON-ответ -> ArduPilot SITL`

Главный инженерный смысл работы состоял в том, чтобы MATLAB перестал
периодически отправлять JSON сам по себе и начал отвечать только на новый
валидный пакет `JSON SITL`.

## Почему прежний путь через ExternalNav не является основным

Официальная документация ArduPilot и локальный пример
`libraries/SITL/examples/JSON/MATLAB` задают базовый режим сопряжения
через JSON backend без обязательного использования внешней навигации по
`MAVLink`.

Поэтому в TASK-25 основной путь был приведен к официальной схеме:

- MATLAB слушает UDP-порт `9002`;
- один валидный двоичный пакет приводит ровно к одному шагу физики;
- один шаг физики приводит ровно к одному JSON-ответу;
- временной шаг задается `frame_rate` из пакета ArduPilot;
- `frame_count` используется для учета дублей, пропусков и перезапуска
  контура.

`ExternalNav` не использовался как основной способ восстановления
работоспособности TASK-25.

## Какие правила официального примера перенесены

В основной путь `universal-mr-uav` перенесены следующие правила
официального MATLAB backend:

- открытие UDP-слушателя на порту `9002`;
- строгий lockstep-цикл `packet -> step -> response`;
- вычисление шага интегрирования:
  `dt = min(1 / frame_rate, max_timestep)`;
- использование `frame_count` для обнаружения дублей и пропусков;
- минимальный официальный состав JSON-полей:
  - `timestamp`;
  - `imu.gyro`;
  - `imu.accel_body`;
  - `position`;
  - `velocity`;
  - `attitude`;
- формирование `JSON` с переводом строки до и после полезной структуры,
  как в официальном примере.

## Формирование accel_body

В TASK-25 `accel_body` приведен к формуле официального MATLAB-примера
ArduPilot.

Сравнение выполнено для пяти характерных состояний:

- покой на земле;
- зависание;
- вертикальное ускорение вверх;
- вертикальное ускорение вниз;
- малый крен и тангаж.

Результат:

- для всех летательных состояний расхождение с официальной формулой
  равно нулю в пределах вычислительной точности;
- максимальное расхождение `9.81 м/с^2` наблюдается только в режиме
  покоя на земле и объясняется отдельным наземным ограничением, которое
  удерживает модель от нештатного проникновения ниже опорной плоскости.

## Выбранный профиль параметров ArduPilot

Для официального JSON backend выбран профиль:

`tools/ardupilot/wsl/task25_official_matlab_json.parm`

Фактически считанные после загрузки параметры:

- `INS_USE = 1`;
- `INS_USE2 = 0`;
- `INS_USE3 = 0`;
- `INS_ENABLE_MASK = 1`;
- `LOG_DISARMED = 1`;
- `SCHED_LOOP_RATE = 150`;
- `AHRS_EKF_TYPE = 10`;
- `GPS1_TYPE = 100`;
- `SIM_MAG1_DEVID = 97539`.

Параметры `GPS_TYPE` и `ARMING_CHECK` в текущей сборке не были считаны по
этому сценарию и в отчет занесены как отсутствующие в результате чтения.

Профиль подтвержден через ранний Windows-side `pymavlink` helper,
подключенный к `udpin:0.0.0.0:14552`.

## Результат официального lockstep-прогона на 20 секунд

Выполнен официальный проверочный прогон на 20 секунд физического
времени.

Получено:

- `valid_rx_count = 3203`;
- `json_response_tx_count = 3202`;
- `missed_frame_count = 0`;
- `duplicate_frame_count = 0`;
- `last_frame_count = 3202`.

Это подтверждает, что в основном режиме MATLAB отвечает только после
принятого валидного пакета и не живет на собственном периодическом
таймере.

## Попытка взведения и первого набора тяги

В отдельном официальном прогоне с тем же backend выполнена попытка:

- дождаться устойчивого обмена;
- перейти в `GUIDED`;
- выполнить `arm`;
- при успешном `arm` выполнить `takeoff 1 m`.

Фактический результат:

- `GUIDED` наблюдался по телеметрии;
- `arm` не выполнен;
- `COMMAND_ACK = 4`;
- первая точная причина отказа:
  `Arm: System not initialised`;
- диапазон `motor_pwm_us = 0..1000 мкс`;
- диапазон `motor_cmd_radps = 0..0 рад/с`;
- изменение высоты не подтверждено.

Следовательно, TASK-25 подтвердил работоспособность официального
MATLAB backend, но не завершил переход к принятию команды взведения.

## Что не является летной валидацией

Полученный результат не является:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением штатной траекторной навигации.

Подтвержден только официальный программный контур обмена `ArduPilot JSON
SITL <-> MATLAB backend` и зафиксирован следующий верхний блокер по
взведению.

## Следующий этап

Следующий этап должен быть направлен не на повторное перестроение
transport-layer, а на инициализацию ArduPilot после загрузки
официального профиля:

- установить, почему после применения `AHRS_EKF_TYPE = 10` и
  `GPS1_TYPE = 100` сохраняется `Arm: System not initialised`;
- проверить готовность EKF/AHRS и момент времени первой попытки `arm`;
- затем повторить `arm` и первый набор тяги уже на официальном backend,
  не отходя от схемы `packet -> step -> response`.
