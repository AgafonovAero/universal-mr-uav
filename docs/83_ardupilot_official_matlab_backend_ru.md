# TASK-25. Официальный MATLAB backend ArduPilot для JSON SITL

## Цель

Цель TASK-25 состояла в приведении контура `ArduPilot JSON SITL <-> MATLAB`
к официальной схеме из документации ArduPilot и локального эталонного
примера `libraries/SITL/examples/JSON/MATLAB`.

Целевой цикл:

`ArduPilot binary packet -> один шаг физики MATLAB -> один JSON-ответ`

Это означает, что MATLAB не должен жить на собственном периодическом
таймере и не должен формировать JSON без нового входного пакета ArduPilot.

## Почему ExternalNav не является основным путем TASK-25

`ExternalNav` не рассматривался как основной способ восстановления
работоспособности, потому что официальный пример ArduPilot MATLAB backend
строится на прямом JSON lockstep-обмене без обязательного внешнего источника
навигации по `MAVLink`.

Поэтому в TASK-25 приоритет был отдан именно приведению основного
`JSON/UDP`-контура к официальной схеме, а не расширению обходных путей.

## Какие правила официального примера перенесены

В проект перенесены следующие правила официального MATLAB backend:

- MATLAB слушает `UDP-порт 9002`;
- каждый валидный двоичный пакет ArduPilot приводит ровно к одному шагу
  модели;
- каждый шаг модели приводит ровно к одному JSON-ответу;
- `dt` вычисляется по `frame_rate`;
- `frame_count` используется для учета дублей, пропусков и сброса состояния
  при переинициализации;
- JSON-кадр строится в составе:
  - `timestamp`;
  - `imu.gyro`;
  - `imu.accel_body`;
  - `position`;
  - `velocity`;
  - `attitude`.

В рамках TASK-25 это реализовано как официальный lockstep-режим, а не как
самостоятельный MATLAB-таймер.

## Формирование accel_body и устранение наземного расхождения

После первой версии PR №25 было обнаружено критическое расхождение с
официальным примером в состоянии покоя на земле:

- ранее в отчете фиксировалось `current = [0 0 0]`;
- официальный пример ArduPilot для того же состояния дает
  `official = [0 0 -9.81]`.

Проверка показала, что проблема находилась не в самом основном пути
формирования JSON, а в сценарии сравнения, который использовал отдельное
диагностическое представление наземного состояния.

После исправления основной официальный режим и сценарий сравнения дают:

- `ground_rest`: `current = [0 0 -9.81]`,
  `official = [0 0 -9.81]`,
  `delta_norm = 0`;
- `hover`: `delta_norm = 0`;
- `vertical_accel_up`: `delta_norm = 0`;
- `vertical_accel_down`: `delta_norm = 0`;
- `small_roll_pitch_hover`: `delta_norm = 0`.

Итог: `imu.accel_body` приведен к официальному MATLAB-примеру ArduPilot для
всех пяти проверочных состояний, включая покой на земле.

Отдельный диагностический режим наземного зануления сохранен только как
вспомогательный режим и не используется в штатных сценариях TASK-25.

## Выбранный профиль параметров ArduPilot

Для официального JSON/MATLAB backend используется профиль:

`tools/ardupilot/wsl/task25_official_matlab_json.parm`

Фактически после загрузки подтверждены:

- `INS_USE = 1`;
- `INS_USE2 = 0`;
- `INS_USE3 = 0`;
- `INS_ENABLE_MASK = 1`;
- `LOG_DISARMED = 1`;
- `SCHED_LOOP_RATE = 150`;
- `AHRS_EKF_TYPE = 10`;
- `GPS1_TYPE = 100`;
- `SIM_MAG1_DEVID = 97539`.

Эти параметры используются как имитационная конфигурация официального
backend ArduPilot, а не как летная конфигурация.

## Результат официального lockstep-прогона на 20 секунд

Повторный 20-секундный прогон в официальном режиме дал:

- `valid_rx_count = 3188`;
- `json_response_tx_count = 3187`;
- `missed_frame_count = 0`;
- `duplicate_frame_count = 0`;
- `invalid_rx_count = 0`;
- `last_frame_count = 3187`;
- `max_altitude_m = 0`.

Это подтверждает:

- MATLAB действительно слушает `9002`;
- JSON формируется только после принятого валидного пакета;
- счетчик кадров растет последовательно;
- дубли и пропуски в этом прогоне отсутствуют.

## Результат попыток arm после 20, 60 и 120 секунд

После исправления `accel_body` выполнен отдельный официальный прогон с тремя
моментами попытки `arm`:

- через `20` секунд непрерывного обмена;
- через `60` секунд;
- через `120` секунд.

Результат:

- после `20` секунд:
  - `arm = нет`;
  - `COMMAND_ACK = 4`;
  - первая причина:
    `Arm: System not initialised`;
- после `60` секунд:
  - `arm = нет`;
  - `COMMAND_ACK = 4`;
  - первая причина:
    `Arm: System not initialised`;
- после `120` секунд:
  - `arm = нет`;
  - `COMMAND_ACK = 0`;
  - в `STATUSTEXT` наблюдалось сообщение `Arming motors`, однако
    устойчивое состояние `armed = true` не подтверждено.

Дополнительно зафиксировано:

- `valid_rx_count = 5855`;
- `json_response_tx_count = 5854`;
- `missed_frame_count = 0`;
- `duplicate_frame_count = 0`;
- `last_frame_count = 5854`;
- `motor_pwm_us = 0..1058 мкс`;
- `motor_cmd_radps = 0..52.2 рад/с`.

Главный вывод: после устранения расхождения по `accel_body` первый
устойчиво наблюдаемый блокер остается следующим:

`Arm: System not initialised`

## A/B-проверка влияния наземного diagnostic ground clamp

Для проверки причинной связи выполнено A/B-сравнение:

- вариант A: `diagnostic_ground_clamp = true`;
- вариант B: `diagnostic_ground_clamp = false`.

В обоих вариантах получено:

- `arm = нет`;
- `COMMAND_ACK = 4`;
- первая причина:
  `Arm: System not initialised`.

Следовательно, diagnostic ground clamp не является причиной текущего отказа
инициализации.

## Что не является летной валидацией

Полученный результат не является:

- летной валидацией;
- подтверждением штатного взлета;
- подтверждением устойчивого автоматического полета;
- подтверждением завершенного цикла `arm -> takeoff`.

Подтверждено только:

- правильное приведение контура к официальной MATLAB-схеме ArduPilot;
- правильное формирование `imu.accel_body` в официальном режиме;
- работоспособность lockstep-обмена;
- наличие следующего реального блокера
  `Arm: System not initialised`.

## Следующий этап

Следующий этап должен быть направлен уже не на `accel_body`, а на анализ
внутренней инициализации ArduPilot после запуска официального MATLAB
backend:

- проверить готовность EKF и AHRS;
- установить, какие подсистемы еще остаются неинициализированными;
- повторить попытку `arm` только после подтверждения завершенной
  инициализации.
