# TASK-19. Диагностика согласованности акселерометров ArduPilot JSON

## Назначение

TASK-19 продолжает практический стенд TASK-18 для режима
`ArduPilot SITL + MATLAB-модель universal-mr-uav + Mission Planner`.

В TASK-18 были подтверждены:

- штатный режим `fly-internal` со взведением и командой `takeoff 5 m`;
- устойчивый `JSON/UDP`-обмен между `ArduPilot` и MATLAB-моделью;
- отказ взведения в JSON-режиме с сообщением `Arm: Accels inconsistent`.

Назначение TASK-19:

- проверить состав и физический смысл инерциальных данных в первых
  `JSON`-кадрах;
- сравнить несколько соглашений по полю `imu.accel_body`;
- проверить влияние нескольких виртуальных ИНС `ArduPilot`;
- проверить влияние задержки перед попыткой `arm`;
- выбрать минимальное инженерное исправление или честно зафиксировать,
  что штатное исправление пока не найдено.

## Что именно означает отказ `Arm: Accels inconsistent`

Сообщение `Arm: Accels inconsistent` в `ArduPilot` относится к проверке
согласованности нескольких акселерометров. На практике это означает, что
внутренний модуль инерциальных датчиков оценивает рассогласование
между экземплярами ИНС выше допустимого порога и запрещает взведение.

Для режима `ArduPilot JSON + MATLAB-модель` это может быть связано с
несколькими причинами:

- неверное физическое толкование `imu.accel_body`;
- рассогласование между несколькими виртуальными ИНС;
- слишком ранняя попытка `arm`, пока `EKF/AHRS` не стабилизировались;
- несогласованность ориентации и ускорения в первых кадрах обмена.

## Выполненные проверки

### 1. Диагностика первых `JSON`-кадров

Сценарий:

- `scripts/diagnose_ardupilot_json_imu_consistency.m`

Сохраняются:

- первые 200 реально отправленных `JSON`-кадров;
- `timestamp`, `imu.gyro`, `imu.accel_body`, `position`, `velocity`,
  `quaternion`;
- `STATUSTEXT` и `COMMAND_ACK` при попытке `arm`.

Подтверждено:

- обязательные поля присутствуют;
- `timestamp` возрастает;
- кватернион конечен и остается нормированным;
- при текущем штатном режиме `json_accel_mode = current`
  первые кадры содержат `imu.accel_body = [0, 0, 0]`.

Это означает, что в покоящемся состоянии MATLAB-модель сейчас передает
в `ArduPilot` нулевую инерциальную перегрузку.

### 2. Опыты по соглашению `imu.accel_body`

Сценарий:

- `scripts/run_ardupilot_json_accel_convention_experiments.m`

Проверены варианты:

- `current`
- `specific_force_frd`
- `minus_specific_force_frd`
- `linear_accel_body`
- `zero_on_ground`

Во всех опытах был сначала восстановлен устойчивый обмен TASK-15:

- `baseline valid_rx_count > 50`
- `baseline response_tx_count > 50`
- `last_frame_count > 0`

После этого выполнялась реальная попытка `arm`.

Результат:

- устойчивый обмен сохранялся во всех вариантах;
- `COMMAND_ACK = 4`;
- `Arm: Accels inconsistent` воспроизводился во всех вариантах;
- переход к `motor_pwm_us > 1000` и `motor_cmd_radps > 0` не произошел.

Следствие:

- простая инверсия знака или замена соглашения `accel_body` сама по себе
  не устраняет отказ взведения.

### 3. Проверка нескольких виртуальных ИНС

Сценарий:

- `scripts/diagnose_ardupilot_ins_instances.m`

Подтверждено в штатном JSON-режиме:

- `INS_USE = 1`
- `INS_USE2 = 1`
- `INS_USE3 = 1`
- поток `SCALED_IMU2` присутствует;
- поток `SCALED_IMU3` присутствует.

Был выполнен отдельный диагностический прогон с профилем:

- `tools/ardupilot/wsl/task19_single_imu_diagnostic.parm`

В нем задавались:

- `INS_USE = 1`
- `INS_USE2 = 0`
- `INS_USE3 = 0`

Результат диагностического профиля одного ИНС:

- устойчивый обмен сохраняется;
- `COMMAND_ACK = 4`;
- `Arm: Accels inconsistent` остается.

Следствие:

- наличие нескольких виртуальных ИНС подтверждено;
- однако простое отключение `INS_USE2/INS_USE3` не устранило отказ;
- значит, несколько ИНС являются подтвержденным диагностическим фактором,
  но не единственным найденным механизмом проблемы.

### 4. Опыты по задержке перед `arm`

Сценарий:

- `scripts/run_ardupilot_json_arm_delay_experiments.m`

Проверены задержки:

- 5 с
- 15 с
- 30 с
- 60 с

Во всех опытах:

- устойчивый обмен был сохранен;
- `COMMAND_ACK = 4`;
- причина отказа оставалась `Arm: Accels inconsistent`.

Следствие:

- поздняя попытка `arm` сама по себе проблему не устраняет.

## Текущий инженерный вывод

По фактическим журналам TASK-19 подтверждено:

- устойчивый `JSON/UDP`-обмен между `ArduPilot` и MATLAB-моделью
  сохранен;
- отказ взведения `Arm: Accels inconsistent` воспроизводится на
  устойчивом обмене;
- простая смена соглашения `accel_body` не дает результата;
- диагностическое отключение `INS_USE2/INS_USE3` не дает результата;
- увеличение задержки перед `arm` до 60 секунд не дает результата.

Это означает, что штатное минимальное исправление причины отказа
в TASK-19 пока не найдено.

## Что не является летной валидацией

TASK-19 не является:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением готовности режима `ArduPilot JSON + MATLAB-модель`
  к штатной эксплуатации.

## Следующий этап

Следующим этапом целесообразно выполнить адресную диагностику именно
внутренней трактовки `imu.accel_body` в `ArduPilot SITL JSON`, а именно:

- сопоставить ожидаемую `specific force` с фактически передаваемым
  содержимым первых кадров;
- проверить, почему при диагностическом профиле одного ИНС сообщение
  `Accels inconsistent` сохраняется;
- собрать дополнительные `STATUSTEXT` и, при необходимости,
  диагностический профиль с минимально необходимыми параметрами ИНС,
  не подменяя штатное решение отключением всех проверок безопасности.
