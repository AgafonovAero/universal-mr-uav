# Автоматизированная подготовка среды ArduPilot SITL

## Назначение

Настоящий документ описывает результат TASK-13.
На данном этапе проект `universal-mr-uav` получает
воспроизводимый набор операторских сценариев подготовки
внешней среды `ArduPilot SITL` для Windows и `WSL`.

Задача TASK-13 состоит не в подтверждении устойчивого
автоматического полета, а в создании безопасной и
проверяемой процедуры подготовки внешних инструментов,
необходимых для следующего этапа проверки обмена по
`JSON` и `UDP`.

Результат этапа должен обеспечить:

- анализ состояния Windows и `WSL`;
- формирование безопасных команд подготовки `WSL`;
- подготовку внешнего каталога `ArduPilot` внутри `WSL`;
- запуск `ArduCopter` в режиме `JSON` через отдельный
  операторский сценарий;
- понятную связь между подготовкой среды и последующим
  запуском `MATLAB`-сценариев ожидания пакета и
  проверки двустороннего обмена.

## Отличие TASK-13 от TASK-12

TASK-12 подтвердил, что проект умеет:

- диагностировать готовность среды;
- формировать команду запуска `ArduPilot SITL`;
- ожидать первый двоичный пакет от `ArduPilot`;
- различать отсутствие приема, исходящую пробную передачу
  и ответную передачу после принятого пакета.

Однако TASK-12 оставлял оператору значительную часть
подготовительных действий в виде текстовых рекомендаций.
TASK-13 переводит эти рекомендации в набор явных
PowerShell- и Bash-сценариев.

Таким образом:

- TASK-12 описывает, что именно требуется проверить;
- TASK-13 описывает, как безопасно подготовить внешнюю среду
  к этим проверкам.

## Почему WSL и ArduPilot не включаются в репозиторий

Репозиторий `universal-mr-uav` предназначен для хранения
математической модели движения, программного расчетного ядра,
подсистемы датчиков, алгоритма оценивания состояния и
средств сопряжения.

Исходные тексты `ArduPilot` не включаются в данный репозиторий,
поскольку:

- они являются внешним программным комплексом управления;
- их жизненный цикл и обновление должны оставаться независимыми;
- включение внешнего дерева исходных текстов резко увеличивает
  объем и сложность сопровождения репозитория;
- оператор может размещать `ArduPilot` в отдельном каталоге
  внутри файловой системы `WSL`, не смешивая его с
  `universal-mr-uav`.

По той же причине `WSL` рассматривается как внешняя
вычислительная среда, а не как часть содержимого проекта.

## Безопасный порядок подготовки среды

Рекомендуемая последовательность действий следующая.

1. Проверить Windows и `WSL` с помощью
   `tools/ardupilot/windows/Test-ArduPilotEnvironment.ps1`.
2. При необходимости подготовить `WSL` с помощью
   `tools/ardupilot/windows/Setup-WSLForArduPilot.ps1`.
3. Подготовить внешний каталог `ArduPilot` внутри выбранного
   дистрибутива `WSL` с помощью
   `tools/ardupilot/windows/Invoke-ArduPilotWslSetup.ps1`.
4. Запустить `ArduCopter` в режиме `JSON` с помощью
   `tools/ardupilot/windows/Start-ArduPilotJsonSitl.ps1`.
5. Из `MATLAB` выполнить
   `scripts/run_ardupilot_wait_for_packet.m`.
6. После подтвержденного приема пакета выполнить
   `scripts/run_ardupilot_json_udp_handshake.m`.

Этот порядок намеренно разделяет:

- подготовку среды Windows;
- подготовку среды `WSL`;
- подготовку внешнего каталога `ArduPilot`;
- запуск внешнего комплекса управления;
- проверку приема пакета в `MATLAB`;
- проверку ответной передачи после принятого пакета.

## Какие действия выполняются автоматически

Сценарии TASK-13 автоматизируют только те действия,
которые допустимы без скрытого изменения системы:

- сбор информации о Windows и `WSL`;
- формирование команд установки и настройки;
- формирование команд клонирования и подготовки `ArduPilot`;
- формирование команд запуска `ArduCopter` в режиме `JSON`;
- печать последовательности действий для оператора;
- сохранение журналов и отчетов.

Автоматическое выполнение возможно только для разрешенных
действий и только при явном параметре `-Execute` в
PowerShell-сценариях.

## Какие действия требуют прав администратора

Права администратора могут потребоваться в Windows
для следующих операций:

- включение `WSL`, если он не установлен;
- установка дистрибутива `Ubuntu`;
- перевод дистрибутива в версию `WSL2`;
- системное обновление компонентов `WSL`.

TASK-13 не выполняет эти действия скрытно.
Если прав администратора нет, сценарий
`Setup-WSLForArduPilot.ps1` не пытается принудительно
изменять систему, а выводит точную команду,
которую оператор должен запустить осознанно от имени
администратора.

Внутри `WSL` права `sudo` могут потребоваться для
установки зависимостей `ArduPilot`. Эти действия также
выполняются только при явном запуске Bash-сценария.

## Где должен располагаться ArduPilot внутри WSL

Рекомендуемый каталог для внешнего дерева исходных текстов
`ArduPilot`:

```bash
~/src/ardupilot
```

Такой выбор обусловлен следующими причинами:

- каталог находится внутри файловой системы `WSL`;
- производительность операций сборки выше, чем в каталогах
  вида `/mnt/c/...` или `/mnt/d/...`;
- дерево `ArduPilot` остается полностью вне репозитория
  `universal-mr-uav`;
- упрощается воспроизводимость документации и сценариев.

Если сценарий подготовки запускается из каталога на
смонтированном диске Windows, он выдает предупреждение и
напоминает о рекомендуемом размещении.

- `scripts/run_ardupilot_json_udp_handshake.m`

## Запуск ArduCopter в режиме JSON

Для запуска `ArduCopter` используется команда вида:

```bash
Tools/autotest/sim_vehicle.py -v ArduCopter -f quad --model JSON:127.0.0.1 --console --map
```

В Bash-сценарии запуска предусмотрены параметры:

- `--no-console`
- `--no-map`
- `--ip`
- `--vehicle`
- `--frame`

В PowerShell-обертке предусмотрены параметры:

- `-DistroName`
- `-NoConsole`
- `-NoMap`
- `-Ip`
- `-Execute`

По умолчанию PowerShell-обертка только печатает команду
и не запускает внешний комплекс управления автоматически.

## Как затем запустить ожидание первого пакета из MATLAB

После подготовки среды и запуска `ArduCopter` оператор
должен перейти в репозиторий `universal-mr-uav` и выполнить:

```matlab
scripts/run_ardupilot_wait_for_packet.m
```

Этот сценарий:

- проверяет состояние среды;
- печатает рекомендуемую команду запуска;
- открывает `UDP`-прием;
- ожидает первый двоичный пакет от `ArduPilot`;
- завершается по тайм-ауту без зависания.

После подтвержденного приема пакета следует выполнить:

```matlab
scripts/run_ardupilot_json_udp_handshake.m
```

Только на этом шаге можно говорить о проверке
ответной передачи после принятого пакета.

## Ограничения текущего этапа

TASK-13 имеет принципиальные ограничения:

- не выполняет летную валидацию;
- не подтверждает устойчивый автоматический полет;
- не изменяет математическую модель движения;
- не изменяет модели `Simulink`;
- не включает исходные тексты `ArduPilot` в репозиторий;
- не устанавливает `WSL` автоматически без явного режима
  выполнения и без прав администратора;
- не заявляет получение двоичного пакета `ArduPilot`,
  если пакет фактически не был принят.

Если среда не готова, это не считается ошибкой этапа.
Этап считается выполненным, если:

- сценарии подготовки созданы;
- они работают в безопасном режиме по умолчанию;
- диагностика среды выполняется без аварии;
- порядок дальнейших действий документирован.

## Критерий перехода к следующему этапу

Переход к следующему этапу возможен после одновременного
выполнения следующих условий:

1. `WSL` установлен и готов к работе.
2. Внутри `WSL` подготовлен внешний каталог `ArduPilot`.
3. `sim_vehicle.py` обнаружен.
4. `ArduCopter` запущен в режиме `JSON`.
5. Сценарий `run_ardupilot_wait_for_packet.m`
   подтвердил прием реального двоичного пакета.
6. Сценарий `run_ardupilot_json_udp_handshake.m`
   подтвердил ответную передачу после принятого пакета.

Только после этого можно переходить к следующему
этапу уточнения параметров сопряжения и длительных
расчетных прогонов с работающим `ArduPilot SITL`.
