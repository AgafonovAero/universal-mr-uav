# Диагностика отказа взведения ArduPilot по частоте обмена

## Назначение документа

Документ фиксирует замысел и фактическое состояние TASK-17, посвященного
диагностике отказа взведения `ArduPilot` при работе стенда
`ArduPilot SITL <-> MATLAB-модель` через `JSON/UDP`.

Основная задача этапа состояла не в изменении математической модели движения,
а в выявлении причин, по которым внешний комплекс управления полетом
`ArduPilot` не переходит к ненулевым командам винтомоторной группы.

## Отличие TASK-17 от TASK-16

В TASK-16 были подтверждены:

- запуск `ArduPilot SITL`;
- обмен двоичными пакетами и строками `JSON`;
- запуск наземной станции `Mission Planner`;
- работоспособность основного контура обмена в пределах контрольного прогона.

TASK-17 должен был ответить на следующий инженерный вопрос:

можно ли поднять частоту обмена и довести стенд до состояния, в котором
`ArduPilot` допускает взведение и выдает ненулевые команды ШИМ.

## Исходная диагностическая гипотеза

После TASK-16 предполагалось, что отказ взведения связан с недостаточной
частотой главного цикла и частотой данных гироскопа, поскольку в предыдущих
журналах фигурировали сообщения вида:

- `Arm: Main loop slow (42Hz < 50Hz)`;
- `Arm: Gyro 0 rate 65Hz < loop rate*1.8 90Hz`.

На этой основе в TASK-17 были предусмотрены:

- измерение фактической частоты обмена;
- оптимизация быстрого цикла `MATLAB`;
- серия опытов с вариантами `SCHED_LOOP_RATE`;
- повторная попытка взведения.

## Используемые варианты параметров

Для TASK-17 подготовлены два дополнительных файла параметров:

- `tools/ardupilot/wsl/task17_loop_rate_50.parm`;
- `tools/ardupilot/wsl/task17_loop_rate_30.parm`.

Назначение файлов:

- `task17_loop_rate_50.parm` — ослабленный штатный режим с `SCHED_LOOP_RATE = 50`;
- `task17_loop_rate_30.parm` — диагностический режим с `SCHED_LOOP_RATE = 30`.

В качестве базового сравнения использовалась конфигурация TASK-16:

- `tools/ardupilot/wsl/task16_arducopter_loop_rate.parm`.

## Повторный измерительный прогон при восстановлении публикации

В ходе восстановления артефактов выяснилось, что ряд ключевых файлов TASK-17
имел нулевой размер. Для восстановления публикации сценарии были повторно
запущены на текущем стенде.

Повторный измерительный прогон показал:

- адрес Windows-хоста для `WSL2`: `172.19.208.1`;
- `valid_rx_count = 1`;
- `json_tx_count = 1792`;
- `response_tx_count = 0`;
- средняя частота исходящих строк `JSON`: `89.6058 Гц`;
- устойчивый входящий поток двоичных пакетов повторно не подтвержден;
- `frame_count` не увеличивается.

Это означает, что при восстановлении публикации была повторно подтверждена
работоспособность запуска и одиночного приема пакета, но не была повторно
подтверждена непрерывная работа обмена на уровне, достаточном для проверки
взведения.

## Повторная серия опытов по взведению

Для трех диагностических конфигураций были получены близкие результаты:

- `valid_rx_count = 1`;
- `response_tx_count = 0`;
- взведение не достигнуто;
- диапазон `motor_pwm_us` остается неопределенным или эквивалентным
  невозбужденному состоянию;
- диапазон `motor_cmd_radps` остается `0..0 рад/с`;
- после опыта процесс `arducopter` не подтверждается как живой.

Для всех трех опытов первой установленной причиной отказа стала запись:

- `[Errno 104] Connection reset by peer`.

## Что подтверждено на текущий момент

Повторной проверкой публикации подтверждено:

- сценарии TASK-17 восстановлены как непустые текстовые файлы;
- журналы TASK-17 восстановлены как непустые файлы UTF-8;
- `MAT` и `CSV` отчеты TASK-17 восстановлены как непустые файлы;
- графики TASK-17 восстановлены как непустые файлы `PNG`;
- `ArduPilot SITL` продолжает запускаться существующим стендом TASK-15;
- серия опытов TASK-17 может быть повторно воспроизведена.

## Что не подтверждено

Повторный прогон при восстановлении публикации не подтверждает:

- устойчивую частоту обмена, достаточную для проверки взведения;
- переход `ArduPilot` в состояние взведения;
- ненулевые команды ШИМ на винтомоторную группу;
- подтвержденный порядок двигателей;
- первый управляемый расчетный прогон.

## Почему документ не заявляет больше, чем подтверждено

Восстановление файлов публикации и инженерная интерпретация результатов должны
опираться только на фактически сохраненные журналы.

Поскольку повторный прогон на текущем состоянии стенда дал только единичный
валидный двоичный пакет и дальнейший разрыв соединения, в документе не
заявляется:

- устранение отказа взведения;
- подтвержденное достижение частоты обмена выше `90 Гц` по входящим пакетам;
- устойчивый автоматический полет;
- летная валидация.

## Следующий инженерный шаг

Следующий шаг должен быть выделен в отдельную задачу и посвящен именно
диагностике причины разрыва соединения `Connection reset by peer` после первого
принятого пакета `JSON/UDP`, а не дальнейшему расширению функциональности
стенда.
