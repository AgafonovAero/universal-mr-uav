# Средство сопряжения с ArduPilot для программного моделирования

## Назначение

Настоящий документ описывает результаты TASK-10, в рамках которого в
репозитории подготовлено средство сопряжения с внешним комплексом
управления полетом `ArduPilot` на уровне расчетного ядра.

Результаты TASK-10 следует рассматривать как подготовительный этап перед
TASK-11. В TASK-11 реализован первый воспроизводимый обмен данными по
`JSON` и `UDP`, однако этот следующий шаг не отменяет ограничений
TASK-10 и не превращает данный документ в описание летной валидации.

Цель этапа состоит в следующем:

- определить состав данных, которые в дальнейшем должны участвовать в
  обмене с `ArduPilot`;
- зафиксировать типовую конфигурацию каналов связи;
- реализовать прозрачное преобразование команд широтно-импульсной
  модуляции в команды частоты вращения винтов;
- провести проверочные замкнутые прогоны без запуска реального
  `ArduPilot SITL`;
- подготовить документы и исходные журналы, необходимые для следующего
  этапа работ.

TASK-10 не является:

- летной валидацией;
- полноценным сопряжением с работающим внешним автопилотом;
- подтверждением готовности к штатной эксплуатации;
- заменой последующего этапа согласования параметров внешнего комплекса
  управления.

## Место в архитектуре проекта

Средство сопряжения TASK-10 размещено в пакете `src/+uav/+ardupilot/` и
использует уже существующие элементы проекта:

- математическую модель движения;
- подсистему датчиков;
- алгоритм оценивания состояния;
- сценарии моделирования на уровне MATLAB;
- тонкие оболочки `Simulink`, не являющиеся главным местом хранения
  физической логики.

На текущем этапе средство сопряжения не изменяет:

- физическую модель объекта управления;
- расчетные коэффициенты объекта управления;
- модуль оценивания состояния;
- модели `models/mil_top.slx` и `models/sil_top.slx`.

Средство сопряжения занимает промежуточное положение между внутренними
данными математической модели и будущим внешним обменом с `ArduPilot`.

## Отличие от исследовательской ветки PR #7

Исследовательская ветка PR `#7` относится к отдельной работе по
внешнему контроллеру `atc_controller`.

TASK-10:

- не зависит от PR `#7`;
- не переносит изменения из ветки `task/08a-atc-mil-bridge`;
- не заменяет исследовательские работы по внешнему контроллеру;
- подготавливает самостоятельное средство сопряжения именно для
  `ArduPilot`.

Таким образом, результаты PR `#7` и TASK-10 относятся к разным
направлениям работ и не должны смешиваться при инженерной оценке.

## Состав модуля сопряжения

В TASK-10 созданы следующие основные компоненты:

- `default_json_config.m` — типовая конфигурация для JSON-обмена;
- `inspect_sitl_environment.m` — проверка локальной готовности среды;
- `pack_json_fdm_packet.m` — формирование канонического пакета данных;
- `unpack_servo_outputs.m` — разбор команд ШИМ по каналам;
- `pwm_to_motor_radps.m` — перевод длительности импульсов в команды
  частоты вращения винтов;
- `validate_json_packet.m` — структурная проверка пакета данных.

Кроме того, в составе TASK-10 имеются:

- вспомогательная функция формирования проверочного пакета команд ШИМ;
- исполнитель сценария моделирования для проверочных замкнутых прогонов;
- сценарии локального запуска для проверки среды и двух типовых
  режимов;
- модульные проверки, подтверждающие корректность состава пакета данных
  и преобразования команд ШИМ.

## Состав пакета данных

Функция `pack_json_fdm_packet.m` подготавливает канонический пакет
данных, который в дальнейшем может быть сериализован в JSON, но на
текущем этапе сам сетевой обмен не выполняется.

В состав пакета включены:

- время моделирования `time_s`;
- положение в земной системе координат `NED`;
- скорость в земной системе координат `NED`;
- скорость в связанной системе координат;
- кватернион ориентации;
- угловые скорости в связанной системе координат;
- измерения инерциального блока;
- барометрические параметры;
- вектор магнитного поля в связанной системе координат;
- данные спутниковой навигации;
- служебная информация о конфигурации обмена.

Такой состав пакета позволяет заранее согласовать структуру данных,
которая потребуется на следующем этапе реального сопряжения.

## Системы координат и обозначения

В TASK-10 применяются те же соглашения, что и в расчетном ядре проекта:

- земная система координат — `NED`;
- связанная система координат — `FRD`, то есть `X` вперед,
  `Y` вправо, `Z` вниз;
- ориентация задается кватернионом в скалярно-первой форме;
- угловые скорости относятся к связанной системе координат;
- линейные скорости в `NED` и в связанной системе координат не
  смешиваются;
- все расчетные величины задаются в системе СИ.

Эти соглашения зафиксированы в составе пакета данных и в конфигурации
средства сопряжения.

## Преобразование команд ШИМ

Функция `pwm_to_motor_radps.m` выполняет прямое и прозрачное
преобразование длительности импульсов ШИМ в команды частоты вращения
винтов.

Преобразование выполняется по следующей схеме:

1. входные значения ограничиваются диапазоном
   `cfg.pwm_min_us ... cfg.pwm_max_us`;
2. ограниченные значения нормируются на интервал `[0, 1]`;
3. нормированное значение переносится в диапазон
   `params.motor.omega_min_radps ... params.motor.omega_max_radps`;
4. итоговые команды еще раз ограничиваются допустимыми пределами.

На текущем этапе это преобразование трактуется как инженерная
подготовка. Окончательное согласование семантики команд ШИМ с реальным
`ArduPilot` должно выполняться отдельно.

## Проверочные замкнутые прогоны

Для TASK-10 подготовлены два проверочных сценария:

- сценарий режима зависания;
- сценарий ступенчатого воздействия по рысканию.

Эти сценарии:

- не запускают реальный `ArduPilot SITL`;
- не используют сетевой обмен по UDP;
- не выполняют реальных переходов режимов внешнего автопилота;
- служат для первичной проверки работоспособности структуры данных и
  преобразования команд.

На каждом шаге исполнитель сценария моделирования:

1. формирует состояние объекта управления;
2. вычисляет измерения подсистемы датчиков;
3. обновляет оценку состояния;
4. подготавливает пакет данных для будущего внешнего обмена;
5. подставляет тестовый пакет команд ШИМ;
6. переводит команды ШИМ в частоты вращения винтов;
7. подает команды на математическую модель движения.

## Проверка локальной среды

Функция `inspect_sitl_environment.m` не пытается имитировать наличие
реального `ArduPilot`, если он отсутствует.

Она только проверяет:

- наличие `WSL`, если проверка допустима на текущей платформе;
- наличие `Python`;
- наличие `sim_vehicle.py` в `PATH`;
- наличие локального каталога `ArduPilot`, если путь задан явно.

Если какой-либо из этих элементов отсутствует, функция возвращает
структурированный результат и не приводит к аварийному завершению
сценария.

## Ограничения текущего этапа

TASK-10 имеет следующие ограничения:

- отсутствует реальный обмен данными по UDP;
- отсутствует действующий экземпляр `ArduPilot SITL`;
- не выполнено полное согласование порядка двигателей для всех режимов;
- не согласованы команды взведения, взлета и переключения режимов;
- не подтверждена устойчивость замкнутого полета с внешним автопилотом;
- летная валидация не проводилась.

Следовательно, результаты TASK-10 следует рассматривать только как
подготовительный инженерный этап перед следующим шагом сопряжения.

## Следующий этап работ

Следующий этап должен включать:

1. подготовку локальной среды с `WSL`;
2. получение рабочего checkout `ArduPilot`;
3. запуск `sim_vehicle.py`;
4. настройку реального обмена данными между `universal-mr-uav` и
   `ArduPilot`;
5. согласование порядка двигателей, диапазонов ШИМ и логики режимов;
6. расширение сценариев верификации после появления реального обмена.

Подробности следующего этапа с реализацией обмена данными по `JSON` и
`UDP` приведены в документе `docs/71_ardupilot_json_udp_ru.md`.
