# Средство сопряжения с ArduPilot для программного моделирования

## Назначение

Настоящий документ описывает результаты TASK-10, в рамках которого в
репозитории подготовлено средство сопряжения с внешним комплексом
управления полетом `ArduPilot` на уровне программно-реализованного
расчетного ядра.

Цель этапа:

- зафиксировать состав данных для последующего обмена с `ArduPilot`;
- задать типовую конфигурацию каналов обмена;
- определить преобразование команд широтно-импульсной модуляции в
  команды частоты вращения винтов;
- провести проверочные замкнутые прогоны без запуска реального
  `ArduPilot SITL`;
- подготовить документацию и исходные журналы прогонов.

TASK-10 не является:

- летной валидацией;
- полнофункциональным сопряжением с работающим внешним автопилотом;
- подтверждением готовности к штатной эксплуатации;
- заменой последующего этапа настройки и согласования параметров.

## Место в архитектуре проекта

Средство сопряжения TASK-10 размещено в пакете `src/+uav/+ardupilot/` и
работает поверх существующего расчетного ядра.

На текущем этапе оно не изменяет:

- математическую модель движения;
- подсистему датчиков;
- алгоритм оценивания состояния;
- тонкие имитационные оболочки `models/mil_top.slx` и `models/sil_top.slx`.

Средство сопряжения используется как промежуточный слой между:

- состоянием и измерениями, формируемыми внутренней моделью;
- будущим внешним обменом данными с `ArduPilot`;
- командами ШИМ, которые в дальнейшем должны поступать от внешнего
  комплекса управления полетом.

Таким образом, TASK-10 подготавливает не внешний полетный комплекс, а
внутреннее проектное описание границы сопряжения.

## Отличие от PR #7

Ветка PR `#7` относится к исследовательскому средству сопряжения с
отдельным внешним контроллером `atc_controller`.

TASK-10:

- не зависит от PR `#7`;
- не переносит изменения из ветки `task/08a-atc-mil-bridge`;
- не заменяет исследовательские работы по `atc_controller`;
- подготавливает отдельное средство сопряжения именно для `ArduPilot`.

## Состав модуля сопряжения

В TASK-10 созданы следующие основные функции:

- `default_json_config.m` - типовая конфигурация каналов обмена и
  параметров ШИМ;
- `inspect_sitl_environment.m` - проверка локальной готовности среды;
- `pack_json_fdm_packet.m` - формирование пакета данных для последующего
  JSON-обмена;
- `unpack_servo_outputs.m` - разбор команд ШИМ по каналам исполнительных
  органов;
- `pwm_to_motor_radps.m` - преобразование длительности импульсов ШИМ в
  команды по частоте вращения винтов;
- `make_loopback_servo_packet.m` - формирование тестового пакета команд
  для проверочного замкнутого прогона;
- `validate_json_packet.m` - структурная проверка корректности пакета
  данных;
- `uav.sim.run_case_with_ardupilot_loopback` - исполнитель сценария
  моделирования для проверочного замкнутого прогона.

## Состав пакета данных

Функция `pack_json_fdm_packet.m` формирует канонический пакет данных,
который предназначен для будущей сериализации в JSON, но сам по себе не
осуществляет сетевой обмен.

В пакет данных включены:

- время моделирования `time_s`;
- положение в земной системе координат `NED`;
- скорость в земной системе координат `NED`;
- кватернион ориентации;
- угловые скорости в связанной системе координат;
- линейные ускорения инерциального измерительного блока в связанной
  системе координат;
- гироскопические измерения;
- барометрическая высота;
- вектор магнитного поля в связанной системе координат;
- данные спутниковой навигации по положению и скорости.

В документируемых соглашениях явно зафиксировано:

- земная система координат - `NED`;
- связанная система координат - `FRD`, то есть `X` вперед,
  `Y` вправо, `Z` вниз;
- скорость в `NED` и скорость в связанной системе координат не должны
  смешиваться;
- кватернион задается в скалярно-первой форме.

## Преобразование команд ШИМ

Функция `pwm_to_motor_radps.m` выполняет прозрачное преобразование
длительности импульсов ШИМ в команды по частоте вращения винтов.

Преобразование выполняется в следующем порядке:

1. входные значения ограничиваются диапазоном
   `cfg.pwm_min_us ... cfg.pwm_max_us`;
2. значения нормируются в диапазон `[0, 1]`;
3. нормированное значение переносится в диапазон
   `params.motor.omega_min_radps ... params.motor.omega_max_radps`;
4. итоговые команды ограничиваются по допустимым пределам.

На текущем этапе это преобразование рассматривается как прозрачная
инженерная заготовка. Полное согласование семантики команд ШИМ с
реальным `ArduPilot` должно выполняться на следующем этапе.

## Проверочные замкнутые прогоны

Для текущего этапа предусмотрены два сценария:

- `scripts/run_ardupilot_loopback_hover.m`;
- `scripts/run_ardupilot_loopback_yaw_step.m`.

Эти сценарии:

- не запускают реальный `ArduPilot SITL`;
- не используют сетевой обмен;
- не выполняют реальные режимные переключения внешнего автопилота;
- служат только для первичной проверки работоспособности проектного
  описания границы сопряжения.

Исполнитель сценария моделирования
`uav.sim.run_case_with_ardupilot_loopback` на каждом шаге:

1. формирует состояние объекта управления;
2. формирует измерения подсистемы датчиков;
3. обновляет алгоритм оценивания состояния;
4. подготавливает пакет данных для будущего внешнего обмена;
5. подставляет тестовый пакет команд ШИМ;
6. переводит команды ШИМ в частоты вращения винтов;
7. передает команды на математическую модель движения.

## Проверка локальной среды

Функция `inspect_sitl_environment.m` не пытается имитировать наличие
реального `ArduPilot`, если он отсутствует.

Она только проверяет:

- наличие `WSL`, если это возможно на текущей платформе;
- наличие `Python`;
- наличие `sim_vehicle.py` в `PATH`, если это возможно;
- наличие локального каталога `ArduPilot`, если путь задан через
  конфигурацию или переменные среды.

Если какой-либо из этих элементов отсутствует, функция возвращает
структурированный отчет и не приводит к аварийному завершению сценария.

## Ограничения текущего этапа

TASK-10 имеет следующие ограничения:

- отсутствует реальный обмен пакетами данных по UDP;
- отсутствует действующий внешний экземпляр `ArduPilot SITL`;
- не выполнено согласование порядка двигателей для всех режимов;
- не выполнено согласование режимов взведения, взлета и переключения
  режимов;
- не подтверждена устойчивость замкнутого полета с внешним автопилотом;
- не проводилась летная валидация.

Следовательно, результаты TASK-10 следует трактовать только как
инженерную подготовку следующего этапа работ.

## Следующий этап работ

Следующий этап должен включать:

1. подготовку локальной среды с `WSL` и рабочим `ArduPilot` checkout;
2. запуск `sim_vehicle.py`;
3. настройку средства обмена данными между `universal-mr-uav` и
   `ArduPilot`;
4. согласование порядка двигателей, диапазонов ШИМ и логики режимов;
5. проверку команд взведения, взлета и удержания режимов;
6. расширение сценариев верификации после появления реального внешнего
   обмена.
