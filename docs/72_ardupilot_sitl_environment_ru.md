# Подготовка рабочей среды ArduPilot SITL и проверка первого пакета

## Назначение

Настоящий документ описывает результат TASK-12. На данном этапе проект
`universal-mr-uav` переходит от внутренней самопроверки средства обмена
данными к воспроизводимой процедуре подготовки внешней среды
`ArduPilot SITL` и подтверждения факта приема первого двоичного пакета
от внешнего программного комплекса.

Инженерная задача TASK-12 состоит не в подтверждении устойчивого
автоматического полета, а в фиксации следующего минимального результата:

- локальная среда анализируется на готовность к запуску `ArduPilot SITL`;
- формируются команды ручной подготовки `WSL` и внешнего каталога
  `ArduPilot`;
- формируется команда запуска `ArduCopter` в режиме `JSON`;
- вводится отдельный сценарий ожидания первого двоичного пакета;
- сценарий двустороннего обмена различает пробную исходящую передачу и
  ответную передачу после принятого пакета.

Таким образом, TASK-12 относится к подготовке воспроизводимой среды
программного моделирования и не относится к летной валидации.

## Отличие TASK-12 от TASK-11

TASK-11 решал задачу первого воспроизводимого обмена данными по `JSON`
и `UDP` на уровне внутренних средств репозитория. На том этапе были
подтверждены:

- кодирование исходящего пакета данных в строку `JSON`;
- разбор двоичного пакета `ArduPilot`;
- работа транспортного уровня `UDP` в среде MATLAB;
- локальная самопроверка без работающего `ArduPilot`.

TASK-12 не меняет математическую модель движения, не меняет алгоритм
оценивания состояния и не меняет модели `Simulink`. Вместо этого он
добавляет недостающий организационный и инструментальный слой:

- проверку готовности `WSL`, `Python`, `Git`, `sim_vehicle.py` и
  локального каталога `ArduPilot`;
- формирование команд подготовки внешней среды;
- отдельный сценарий ожидания первого двоичного пакета;
- строгую семантику статусов обмена;
- отчетливое различие между отсутствием приема и подтвержденной
  ответной передачей.

Если TASK-11 отвечал на вопрос о работоспособности внутреннего средства
обмена, то TASK-12 отвечает на вопрос о том, готова ли внешняя среда к
получению первого реального пакета от `ArduPilot SITL`.

## Подготовка WSL

Если проект запускается в среде Windows, рекомендуемая рабочая схема
для `ArduPilot SITL` основана на использовании `WSL`. В рамках TASK-12
репозиторий не устанавливает `WSL` автоматически. Вместо этого
формируется перечень команд, которые оператор может выполнить осознанно.

Типовая последовательность действий включает:

1. проверку наличия `WSL` командой `wsl --status`;
2. просмотр списка установленных дистрибутивов командой `wsl -l -v`;
3. установку `WSL`, если он отсутствует;
4. обновление среды `WSL`, если требуется;
5. выбор дистрибутива, в котором будет размещен внешний каталог
   `ArduPilot`.

Функция `inspect_sitl_environment.m` и сценарий
`scripts/print_ardupilot_sitl_setup_instructions.m` не выполняют эти
действия автоматически. Они только фиксируют:

- обнаружен ли `WSL`;
- удалось ли получить список дистрибутивов;
- задан ли предпочтительный дистрибутив через
  `cfg.wsl_distro_name`;
- каких элементов не хватает для продолжения работ.

Такой подход исключает скрытое изменение системы и оставляет оператору
контроль над внешней вычислительной средой.

## Получение ArduPilot вне репозитория

Исходные тексты `ArduPilot` не должны включаться в репозиторий
`universal-mr-uav`. Для TASK-12 это правило остается обязательным.

Средство сопряжения использует только:

- локальный путь `cfg.ardupilot_root`, если он задан оператором;
- переменные окружения `ARDUPILOT_ROOT`, `ARDUPILOT_HOME`,
  `ARDUPILOT_DIR`, если они определены;
- наличие `sim_vehicle.py` в `PATH`;
- команды, которые оператор может выполнить во внешнем каталоге.

Типовая последовательность подготовки внешнего каталога включает:

1. выбор рабочего каталога вне `universal-mr-uav`;
2. получение `ArduPilot` из внешнего репозитория;
3. инициализацию подмодулей;
4. подготовку зависимостей в среде `WSL`;
5. сборку `SITL`.

Все эти команды формируются функцией
`make_sitl_setup_commands.m` и выводятся сценарием
`scripts/print_ardupilot_sitl_setup_instructions.m`.

## Формирование команды запуска ArduCopter в режиме JSON

Команда запуска строится функцией
`src/+uav/+ardupilot/make_sitl_start_command.m`.

В команде учитываются:

- тип аппарата `cfg.sitl_vehicle`;
- схема рамы `cfg.sitl_frame`;
- адрес `JSON`-обмена, формируемый из `cfg.udp_local_ip`;
- возможность запуска через `WSL`, если указан дистрибутив.

На этапе TASK-12 формируется именно команда ручного запуска, а не
автоматический запуск `ArduPilot`. Это необходимо по двум причинам:

- запуск внешнего комплекса управления должен оставаться осознанным
  действием оператора;
- в разных конфигурациях Windows и `WSL` адрес узла Windows может
  отличаться от `127.0.0.1`.

Следовательно, сформированная команда рассматривается как инженерная
инструкция, а не как доказательство готовности среды сама по себе.

## Порядок проверки приема двоичного пакета

Для подтверждения приема первого реального пакета введена функция
`src/+uav/+ardupilot/wait_for_sitl_output_packet.m`.

Ее логика такова:

1. проверить доступность транспортного уровня `UDP`;
2. открыть средство обмена с параметрами из конфигурации;
3. ожидать входящий двоичный пакет в течение
   `cfg.udp_handshake_timeout_s`;
4. при получении попытаться разобрать пакет через
   `decode_sitl_output_packet.m`;
5. завершиться без зависания по тайм-ауту, если пакет не поступил.

Ключевое ограничение этого сценария состоит в том, что он не отправляет
строку `JSON` до приема пакета. Это позволяет использовать его как
чистую проверку факта входящего обмена со стороны `ArduPilot`.

Верхнеуровневый сценарий
`scripts/run_ardupilot_wait_for_packet.m` выводит:

- готовность среды;
- команду запуска `ArduPilot`;
- факт приема или неприема пакета;
- значения `magic`, частоты кадров, счетчика кадров и первых четырех
  каналов `ШИМ`, если пакет действительно получен.

## Различие между исходящей пробной передачей и ответной передачей

Одним из главных требований TASK-12 является строгая семантика обмена.
В документации, отчетах и диагностике используются только следующие
формулировки.

### Прием не подтвержден

Этот статус означает, что входящий двоичный пакет от `ArduPilot` не был
получен либо был получен, но не разобран. В таком состоянии нельзя
утверждать, что двусторонний обмен подтвержден.

### Исходящая пробная передача

Этот статус используется только в том случае, когда строка `JSON`
сформирована и отправлена, но входящий двоичный пакет от `ArduPilot`
до этого не был принят и разобран.

Следовательно, пробная исходящая передача подтверждает:

- работоспособность формирования строки `JSON`;
- работоспособность локального отправителя `UDP`.

Она не подтверждает:

- прием со стороны `ArduPilot`;
- успешную обратную реакцию `ArduPilot`;
- ответную передачу после принятого пакета.

### Ответная передача

Этот статус допускается только тогда, когда:

1. входящий двоичный пакет от `ArduPilot` был реально получен;
2. пакет был успешно разобран;
3. после этого была отправлена строка `JSON`.

Именно такая последовательность рассматривается как первый подтвержденный
двусторонний обмен на текущем этапе.

## Ограничения текущего этапа

TASK-12 имеет следующие ограничения:

- не подтверждается устойчивый автоматический полет;
- не выполняется летная валидация;
- не внедряется `MAVLink`;
- не изменяется математическая модель движения;
- не изменяются модели `Simulink`;
- не выполняется автоматическая установка `WSL`;
- не выполняется автоматическое получение исходных текстов `ArduPilot`;
- отсутствие входящего пакета от `ArduPilot` не подменяется
  искусственными данными.

Если реальный пакет не получен, это не считается ошибкой задачи.
В этом случае задача считается выполненной при условии, что:

- среда корректно проанализирована;
- команды подготовки сформированы;
- сценарий ожидания пакета корректно завершается по тайм-ауту;
- сценарий handshake честно фиксирует отсутствие подтвержденного приема.

## Критерий перехода к следующему этапу

Следующий этап можно считать обоснованным только после выполнения
следующих условий:

1. `WSL` подготовлен и выбран рабочий дистрибутив;
2. внешний каталог `ArduPilot` получен и собран вне репозитория
   `universal-mr-uav`;
3. `sim_vehicle.py` доступен;
4. команда запуска `ArduCopter` в режиме `JSON` сформирована и
   воспроизводима;
5. сценарий `run_ardupilot_wait_for_packet.m` подтвердил прием первого
   двоичного пакета;
6. сценарий `run_ardupilot_json_udp_handshake.m` подтвердил именно
   ответную передачу после принятого пакета.

Только после этого имеет смысл переходить к более длинным расчетным
прогонам, согласованию порядка двигателей и дальнейшему развитию
программного моделирования с внешним комплексом управления полетом.

## Связь с TASK-13

TASK-13 развивает результаты настоящего этапа и переводит их из уровня
ручных инженерных рекомендаций в уровень воспроизводимых операторских
сценариев подготовки среды Windows и WSL.

Для этого в следующем этапе вводятся:

- PowerShell-сценарий проверки Windows и `WSL`;
- PowerShell-сценарий безопасной подготовки `WSL`;
- PowerShell-обертка запуска Bash-подготовки `ArduPilot` внутри `WSL`;
- Bash-сценарий подготовки внешнего каталога `ArduPilot`;
- Bash-сценарий запуска `ArduCopter` в режиме `JSON`;
- MATLAB-сценарий общей проверки подготовки среды.

Таким образом, TASK-12 задает диагностическую и методическую основу,
а TASK-13 автоматизирует операторскую подготовку среды без изменения
математической модели движения и без включения исходных текстов
`ArduPilot` в репозиторий `universal-mr-uav`.
