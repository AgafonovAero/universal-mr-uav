# Средство обмена данными с ArduPilot через JSON и UDP

## Назначение

Настоящий документ описывает результаты TASK-11, в рамках которого в
репозитории `universal-mr-uav` реализован первый воспроизводимый обмен
данными с внешним комплексом управления полетом `ArduPilot` по `JSON` и
`UDP`.

Цель этапа состоит в следующем:

- сформировать строку `JSON` из расчетного пакета данных объекта
  управления;
- разобрать двоичный выходной пакет `ArduPilot SITL`;
- реализовать транспортный уровень `UDP` в среде MATLAB;
- подтвердить работоспособность обмена на локальной самопроверке без
  установки `ArduPilot`;
- выполнить корректную попытку обмена с уже запущенным `ArduPilot SITL`,
  если локальная среда к этому готова.

TASK-11 не ставит целью:

- подтверждение устойчивого автоматического полета;
- настройку режимов взведения, взлета, посадки и переключения режимов;
- реализацию протокола `MAVLink`;
- перенос физики объекта управления в `Simulink`;
- летную валидацию.

## Отличие TASK-11 от TASK-10

TASK-10 подготовил структуру средства сопряжения:

- типовую конфигурацию;
- состав пакета данных;
- преобразование длительности импульсов ШИМ в команды частоты вращения
  винтов;
- проверочные замкнутые прогоны без реального сетевого обмена.

TASK-11 развивает этот подготовительный этап и вводит:

- кодирование пакета данных в текст `JSON`;
- разбор двоичного выходного пакета `ArduPilot`;
- открытие, использование и закрытие средства обмена по `UDP`;
- проверочный сценарий локального обмена без `ArduPilot`;
- отдельный сценарий краткой попытки обмена с уже запущенным
  `ArduPilot SITL`.

Таким образом, TASK-11 является развитием TASK-10, а не его заменой.
Ограничения подготовительного этапа сохраняются, пока не подтверждено
длительное устойчивое взаимодействие с работающим внешним комплексом
управления.

## Место в архитектуре проекта

Средство обмена TASK-11 размещено в пакете `src/+uav/+ardupilot/` и
использует уже существующие элементы расчетного ядра:

- математическую модель движения;
- подсистему датчиков;
- алгоритм оценивания состояния;
- средства формирования расчетных сценариев на уровне MATLAB;
- тонкие имитационные оболочки `Simulink`, которые не содержат ручной
  реализации физики объекта управления.

Ключевой принцип сохраняется прежним: исходное достоверное описание
модели находится в `.m`-коде и текстовых файлах, а `Simulink` остается
только средством оркестрации.

## Состав модуля сопряжения

В TASK-11 добавлены или расширены следующие функции:

- `default_json_config.m` — типовая конфигурация каналов связи и
  форматов двоичного пакета;
- `encode_json_fdm_text.m` — кодирование расчетного пакета данных в
  строку `JSON`;
- `decode_sitl_output_packet.m` — разбор двоичного пакета выходов
  `ArduPilot`;
- `json_udp_is_available.m` — проверка доступности `UDP` в текущей
  версии MATLAB;
- `json_udp_open.m` — открытие средства обмена;
- `json_udp_step.m` — один шаг приема и передачи данных;
- `json_udp_close.m` — корректное закрытие средства обмена;
- `make_sitl_start_command.m` — формирование рекомендуемой команды
  запуска `ArduPilot` в режиме `JSON`;
- `run_case_with_ardupilot_udp.m` — расчетный сценарий обмена, который
  использует реальный ответ `ArduPilot`, если он действительно получен.

Дополнительно созданы сценарии верхнего уровня:

- `scripts/run_ardupilot_json_udp_selfcheck.m`;
- `scripts/run_ardupilot_json_udp_handshake.m`.

## Состав входного JSON для ArduPilot

Функция `encode_json_fdm_text.m` формирует строку `JSON` из структуры,
которую подготавливает `pack_json_fdm_packet.m`.

В состав передаваемых данных входят:

- `timestamp` — время моделирования;
- `imu.gyro` — угловые скорости в связанной системе координат;
- `imu.accel_body` — кажущиеся ускорения в связанной системе координат;
- `position` — положение в земной системе координат `NED`;
- `velocity` — скорость в земной системе координат `NED`;
- `quaternion` — кватернион ориентации;
- `body_rates` — угловые скорости корпуса;
- `baro.altitude_m` — барометрическая высота, если она доступна;
- `mag.body_uT` — вектор магнитного поля в связанной системе координат,
  если он доступен;
- `gnss.position_ned_m` и `gnss.velocity_ned_mps` — спутниковые данные,
  если они доступны.

Строка `JSON` завершается символом перевода строки. Сама функция не
выполняет передачу по сети и используется только для формирования
текстового представления пакета данных.

## Состав выходного двоичного пакета от ArduPilot

Функция `decode_sitl_output_packet.m` поддерживает два формата:

1. вариант на 16 каналов:
   - `uint16 magic = 18458`;
   - `uint16 frame_rate`;
   - `uint32 frame_count`;
   - `uint16 pwm[16]`;
2. вариант на 32 канала:
   - `uint16 magic = 29569`;
   - `uint16 frame_rate`;
   - `uint32 frame_count`;
   - `uint16 pwm[32]`.

Для квадрокоптерной схемы текущего проекта из полного массива ШИМ
используются первые четыре канала или порядок, заданный через
`cfg.motor_order`.

Результат разбора оформляется структурой с полями:

- `valid`;
- `magic`;
- `frame_rate_hz`;
- `frame_count`;
- `pwm_us`;
- `motor_pwm_us`;
- `message`.

Если пакет пуст, слишком короток или содержит неизвестное значение
`magic`, функция возвращает `valid = false` и понятное пояснение без
аварийного завершения.

## Системы координат и обозначения

В TASK-11 сохраняются общие соглашения проекта:

- земная система координат — `NED`;
- связанная система координат — `FRD`, то есть `X` вперед, `Y` вправо,
  `Z` вниз;
- ориентация передается кватернионом;
- линейные скорости в `NED` и в связанной системе координат не
  смешиваются;
- угловые скорости относятся к связанной системе координат;
- все расчетные величины записываются в системе СИ.

Эти обозначения должны быть явно сохранены и на следующем этапе, когда
обмен с реальным `ArduPilot SITL` станет длительным и двусторонним.

## Преобразование длительности импульсов ШИМ

Функция `pwm_to_motor_radps.m` переводит длительность импульсов ШИМ в
команды частоты вращения винтов по прозрачному линейному закону:

1. входные значения ограничиваются диапазоном
   `cfg.pwm_min_us ... cfg.pwm_max_us`;
2. ограниченные значения нормируются на интервал `[0, 1]`;
3. нормированное значение переносится в диапазон
   `params.motor.omega_min_radps ... params.motor.omega_max_radps`;
4. итоговая команда еще раз ограничивается допустимыми пределами.

На данном этапе это инженерное средство согласования сигналов, а не
окончательно подтвержденная семантика команд конкретного экземпляра
`ArduPilot`.

## Проверочный обмен без ArduPilot

Сценарий `scripts/run_ardupilot_json_udp_selfcheck.m` предназначен для
локальной самопроверки без установленного `ArduPilot`.

Сценарий подтверждает:

- доступность `UDP` в MATLAB;
- корректность кодирования расчетного пакета в строку `JSON`;
- корректность разбора искусственно сформированного двоичного пакета;
- работоспособность преобразования длительности импульсов ШИМ в команды
  частоты вращения винтов;
- локальный проверочный обмен по `UDP`, если механизм `udpport`
  доступен.

Этот сценарий не выдает себя за реальный обмен с внешним автопилотом и
не подменяет отсутствие `ArduPilot`.

## Проверка локальной среды и попытка обмена с ArduPilot

Сценарий `scripts/run_ardupilot_json_udp_handshake.m` выполняет короткую
попытку обмена с уже запущенным `ArduPilot JSON SITL`.

Если среда не готова, сценарий:

- не завершает работу с ошибкой;
- фиксирует отсутствие готовности;
- сообщает, что пакет от `ArduPilot` не получен;
- выводит рекомендуемую команду запуска, сформированную функцией
  `make_sitl_start_command.m`.

Если двоичный пакет получен, сценарий:

- разбирает его;
- выводит значение `magic`, частоту кадров, номер кадра и первые четыре
  значения ШИМ;
- формирует ответную строку `JSON`;
- отправляет ее по `UDP`;
- записывает итоговый статус обмена.

## Подготовка к проверке с ArduPilot

Для следующего этапа понадобятся:

- рабочая среда `WSL`, если используется Windows;
- установленный `Python`;
- доступный `sim_vehicle.py`;
- локальный checkout `ArduPilot`;
- понимание сетевого адреса, по которому `ArduPilot` ожидает обмен
  `JSON`;
- согласование порядка двигателей и диапазонов ШИМ.

Рекомендуемая команда запуска формируется функцией
`make_sitl_start_command.m` и содержит модель вида
`JSON:<адрес>`.

## Ограничения текущего этапа

TASK-11 имеет следующие ограничения:

- не подтвержден устойчивый автоматический полет;
- отсутствует согласование режимов взведения, взлета и переключения
  режимов;
- не реализован `MAVLink`;
- не выполнено длительное программное моделирование с работающим
  `ArduPilot SITL`;
- летная валидация не проводилась;
- при отсутствии готовой среды сценарий обмена честно фиксирует
  отсутствие пакета от `ArduPilot`.

Следовательно, TASK-11 следует рассматривать как первый воспроизводимый
этап обмена данными, а не как завершенное средство сопряжения,
пригодное к штатной эксплуатации.

## Следующий этап работ

Следующий этап должен включать:

1. подготовку рабочей среды `WSL` и локального checkout `ArduPilot`;
2. запуск `sim_vehicle.py` в режиме `JSON`;
3. подтверждение длительного двустороннего обмена по `UDP`;
4. согласование порядка двигателей, диапазонов ШИМ и семантики команд;
5. расширение расчетных сценариев с использованием реального ответа
   внешнего комплекса управления;
6. отдельную верификацию устойчивости поведения до перехода к более
   сложным режимам и к последующей летной валидации.
