# Средство обмена данными с ArduPilot через JSON и UDP

## Назначение

Настоящий документ описывает результаты TASK-11, в рамках которой
репозиторий `universal-mr-uav` расширен от подготовительной заготовки
средства сопряжения до первого воспроизводимого уровня обмена данными
с `ArduPilot` по `JSON` и `UDP`.

Главная инженерная цель этапа состоит в следующем:

- реализовать формирование исходящего пакета данных в текстовом формате
  `JSON`;
- реализовать разбор входного двоичного пакета, который используется
  `ArduPilot JSON SITL`;
- реализовать транспортный уровень `UDP` в среде MATLAB;
- выполнить локальную самопроверку без установленного `ArduPilot`;
- выполнить короткую попытку обмена с уже работающим `ArduPilot SITL`,
  если среда готова.

При этом TASK-11 не ставит целью:

- подтверждение устойчивого автоматического полета;
- летную валидацию;
- настройку режимов взведения, взлета, посадки и переключения режимов;
- внедрение `MAVLink`;
- перенос физики объекта управления внутрь `Simulink`.

Иными словами, на данном этапе подтверждается только работоспособность
средства обмена данными и прозрачность его интерфейсов.

## Отличие TASK-11 от TASK-10

TASK-10 решала подготовительную задачу. На том этапе были заданы:

- типовая конфигурация средства сопряжения;
- состав пакета данных математической модели движения;
- преобразование длительности импульса ШИМ в команду частоты вращения
  винта;
- проверочный замкнутый прогон без реального сетевого обмена.

TASK-11 развивает этот результат и добавляет уже не только описание
интерфейса, но и первый фактический обмен по сети:

- кодирование пакета данных в строку `JSON`;
- разбор двоичного пакета выходов `ArduPilot`;
- открытие, использование и закрытие `UDP`-средства обмена;
- самопроверку локального обмена без работающего `ArduPilot`;
- короткую попытку реального обмена с уже запущенным `ArduPilot SITL`.

Таким образом, TASK-10 отвечает на вопрос о составе данных и
подготовке интерфейса, а TASK-11 отвечает на вопрос о первом
воспроизводимом уровне обмена через `UDP`.

## Место в архитектуре проекта

Средство обмена TASK-11 размещено в пакете `src/+uav/+ardupilot/`.

Оно использует уже существующие части расчетного ядра:

- математическую модель движения объекта управления;
- подсистему датчиков;
- алгоритм оценивания состояния;
- расчетные сценарии верхнего уровня на языке MATLAB.

Архитектурный принцип проекта остается неизменным:

- исходное достоверное описание модели остается в `.m`-коде и текстовых
  файлах;
- `Simulink` допускается только как тонкая имитационная оболочка;
- физика объекта управления не переносится в схемы `Simulink`;
- средство обмена с внешним программным комплексом управления полетом
  рассматривается как внешний интерфейс над существующим расчетным
  ядром.

## Состав средства обмена

В TASK-11 добавлены или расширены следующие функции:

- `default_json_config.m` — типовая конфигурация обмена по `JSON` и `UDP`;
- `encode_json_fdm_text.m` — кодирование пакета данных в строку `JSON`;
- `decode_sitl_output_packet.m` — разбор двоичного пакета `ArduPilot`;
- `json_udp_is_available.m` — проверка доступности механизма `UDP`
  в текущей версии MATLAB;
- `json_udp_open.m` — открытие средства обмена;
- `json_udp_step.m` — один шаг приема и передачи;
- `json_udp_close.m` — безопасное закрытие средства обмена;
- `make_sitl_start_command.m` — формирование рекомендуемой команды
  запуска `ArduPilot` в режиме `JSON`;
- `run_case_with_ardupilot_udp.m` — расчетный сценарий пошагового обмена
  с использованием существующей математической модели движения.

Дополнительно созданы сценарии верхнего уровня:

- `scripts/run_ardupilot_json_udp_selfcheck.m`;
- `scripts/run_ardupilot_json_udp_handshake.m`.

## Состав исходящего пакета JSON

Функция `encode_json_fdm_text.m` формирует строку `JSON` из структуры,
которую подготавливает `pack_json_fdm_packet.m`.

В состав исходящего пакета включены следующие группы данных:

- `timestamp` — текущее модельное время;
- `imu.gyro` — угловые скорости в связанной системе координат;
- `imu.accel_body` — кажущиеся ускорения в связанной системе координат;
- `position` — положение в земной системе координат `NED`;
- `velocity` — скорость в земной системе координат `NED`;
- `quaternion` — кватернион ориентации;
- `body_rates` — угловые скорости корпуса;
- `baro.altitude_m` — барометрическая высота, если она доступна;
- `mag.body_uT` — вектор магнитного поля в связанной системе координат,
  если он доступен;
- `gnss.position_ned_m` и `gnss.velocity_ned_mps` — спутниковые данные,
  если они доступны.

Строка `JSON` завершается символом перевода строки. Функция
`encode_json_fdm_text.m` только формирует текстовое представление пакета
данных и не выполняет сетевую передачу.

## Состав входного двоичного пакета ArduPilot

Функция `decode_sitl_output_packet.m` поддерживает два формата двоичного
пакета выходов `ArduPilot`.

Первый формат соответствует 16 каналам:

1. `uint16 magic = 18458`;
2. `uint16 frame_rate`;
3. `uint32 frame_count`;
4. `uint16 pwm[16]`.

Второй формат соответствует 32 каналам:

1. `uint16 magic = 29569`;
2. `uint16 frame_rate`;
3. `uint32 frame_count`;
4. `uint16 pwm[32]`.

Результат разбора оформляется структурой со следующими полями:

- `valid`;
- `magic`;
- `frame_rate_hz`;
- `frame_count`;
- `pwm_us`;
- `motor_pwm_us`;
- `message`.

Для текущей конфигурации квадрокоптера используются первые четыре
канала либо порядок, заданный через `cfg.motor_order`.

Если пакет пуст, слишком короток или содержит неизвестное значение
`magic`, функция не завершает работу аварийно, а возвращает
`valid = false` и поясняющее сообщение.

## Системы координат и обозначения

В TASK-11 сохраняются общие соглашения проекта:

- земная система координат — `NED`;
- связанная система координат — `FRD`;
- внутренняя ориентация передается кватернионом;
- линейные скорости в `NED` и в связанной системе координат не
  смешиваются;
- угловые скорости относятся к связанной системе координат;
- расчетные величины записываются в единицах СИ.

Эти соглашения особенно важны для формирования исходящего пакета данных.
На текущем этапе средство обмена должно быть однозначным именно по
системам координат и обозначениям, даже если реальный обмен с
работающим внешним программным комплексом еще не подтвержден.

## Преобразование длительности импульсов ШИМ

Преобразование длительности импульса ШИМ в команду частоты вращения винта
выполняется функцией `pwm_to_motor_radps.m`.

Преобразование включает следующие шаги:

1. ограничение входного сигнала диапазоном `cfg.pwm_min_us` ...
   `cfg.pwm_max_us`;
2. нормирование ограниченного сигнала на интервал `[0, 1]`;
3. перенос нормированного сигнала в диапазон
   `params.motor.omega_min_radps` ...
   `params.motor.omega_max_radps`;
4. повторное ограничение итоговой команды допустимыми пределами.

На данном этапе это инженерное согласование сигналов между средством
обмена и математической моделью движения. Полная семантика команд
конкретной конфигурации `ArduPilot` будет подтверждаться на следующем
этапе, когда появится реальный двусторонний обмен.

## Проверка без работающего ArduPilot

Сценарий `scripts/run_ardupilot_json_udp_selfcheck.m` предназначен для
локальной самопроверки без установленного `ArduPilot`.

Он подтверждает:

- доступность `UDP` в MATLAB;
- корректность кодирования строки `JSON`;
- корректность разбора искусственно сформированного двоичного пакета;
- корректность преобразования длительности импульсов ШИМ;
- корректность локального проверочного обмена по `UDP`.

При этом сценарий самопроверки не выдает себя за реальный обмен с
внешним программным комплексом управления полетом. Искусственный
двоичный пакет используется только здесь и только для внутренней
проверки механизма разбора.

## Попытка обмена с работающим ArduPilot

Сценарий `scripts/run_ardupilot_json_udp_handshake.m` выполняет короткую
попытку обмена с уже запущенным `ArduPilot JSON SITL`.

Логика сценария разделяется на два случая.

Если входной двоичный пакет не получен:

- фиксируется отсутствие пакета от `ArduPilot`;
- формируется и отправляется только исходящая пробная строка `JSON`;
- формируется рекомендуемая команда запуска `ArduPilot`;
- сценарий завершает работу без аварии.

Если входной двоичный пакет получен и разобран:

- фиксируются `magic`, частота кадров, номер кадра и значения ШИМ;
- формируется ответная строка `JSON`;
- выполняется ответная передача по `UDP`;
- статус обмена сохраняется в журнале прогона.

На момент выполнения TASK-11 реальный двоичный пакет от `ArduPilot`
получен не был. Поэтому подтвержден только факт исходящей пробной
передачи, а ответная передача после принятого пакета не проверялась.

## Ограничения текущего этапа

TASK-11 имеет следующие ограничения:

- устойчивый автоматический полет не подтвержден;
- летная валидация не проводилась;
- режимы взведения, взлета, посадки и переключения режимов не
  согласованы;
- `MAVLink` не реализован;
- длительный двусторонний обмен с работающим `ArduPilot SITL` не
  подтвержден;
- при отсутствии готовой среды сценарий реального обмена честно
  фиксирует отсутствие принятого двоичного пакета.

Следовательно, TASK-11 следует рассматривать как этап проверки
интерфейса и транспортного уровня, а не как завершенное средство
сопряжения, пригодное к штатной эксплуатации.

## Следующий этап работ

Следующий этап должен включать:

1. подготовку рабочей среды `WSL`, если используется Windows;
2. получение рабочего checkout `ArduPilot`;
3. запуск `sim_vehicle.py` в режиме `JSON`;
4. подтверждение устойчивого двустороннего обмена по `UDP`;
5. согласование порядка двигателей и диапазонов ШИМ;
6. отдельную проверку длительного программного моделирования перед
   переходом к более сложным режимам и дальнейшей валидации.
