# Диагностика отказа взведения ArduPilot по частоте гироскопа

## 1. Назначение документа

Документ фиксирует результаты TASK-22 по устранению отказа взведения
`Arm: Gyro 0 rate ... < loop rate*1.8 ...` в режиме сопряжения
`ArduPilot JSON + MATLAB-модель` через `JSON/UDP`.

Рассматривается только инженерная диагностика предполетной проверки
ArduPilot и минимальное параметрическое исправление, необходимое для
принятия команды взведения. Летная валидация в рамках данного этапа не
выполнялась.

## 2. Исходная проблема после TASK-21

После снятия блокера `Arm: 3D Accel calibration needed` штатной простой
калибровкой акселерометра устойчивый обмен с внешней математической
моделью сохранился, однако команда `arm` по-прежнему отклонялась.

Фактический блокер имел вид:

`Arm: Gyro 0 rate 430Hz < loop rate*1.8 720Hz`

Это означает, что ArduPilot сравнивает фактическую частоту поступления
инерциальных данных гироскопа с минимально допустимым значением,
равным `1.8 * SCHED_LOOP_RATE`. При `SCHED_LOOP_RATE = 400 Гц` требуется
не менее `720 Гц`, тогда как в текущем JSON-режиме наблюдается около
`430 Гц`.

## 3. Что показала диагностика частот

Сценарий `scripts/diagnose_ardupilot_gyro_loop_rate.m` подтвердил:

- `SCHED_LOOP_RATE = 400 Гц`;
- `SIM_RATE_HZ = 1200 Гц`;
- фактическая частота гироскопа по сообщению ArduPilot: `430 Гц`;
- требуемый минимум: `720 Гц`;
- максимальное допустимое значение `SCHED_LOOP_RATE` при такой частоте:
  `floor(430 / 1.8) = 238 Гц`;
- устойчивый обмен при этом сохраняется:
  `valid_rx_count = 632`,
  `response_tx_count = 631`,
  `last_frame_count = 1129`.

Следовательно, исходная причина отказа заключалась не в потере
`JSON/UDP`-обмена, а в несоответствии частоты главного цикла ArduPilot
фактической частоте выдачи данных гироскопа в режиме внешней модели.

## 4. Проверенные профили главного цикла

Для проверки были подготовлены профили:

- `tools/ardupilot/wsl/task22_sched_loop_250.parm`;
- `tools/ardupilot/wsl/task22_sched_loop_200.parm`;
- `tools/ardupilot/wsl/task22_sched_loop_100.parm`.

Каждый профиль включал базовые исправления предыдущих задач:

- `INS_USE = 1`;
- `INS_USE2 = 0`;
- `INS_USE3 = 0`;
- `INS_ENABLE_MASK = 1`;
- `LOG_DISARMED = 1`.

Сценарий `scripts/verify_ardupilot_sched_loop_profiles.m` показал, что
значения `SCHED_LOOP_RATE = 250`, `200` и `100` фактически применяются
после загрузки ArduPilot. На всех профилях устойчивый `JSON/UDP`-обмен
сохраняется.

## 6. Выбранное исправление

Для основного пути TASK-22 выбран профиль:

- `tools/ardupilot/wsl/task22_sched_loop_200.parm`

с ключевым параметром:

- `SCHED_LOOP_RATE = 200`.

Причина выбора:

- это наибольшее значение `SCHED_LOOP_RATE`, которое снимает блокер по
  частоте гироскопа при наблюдаемой частоте около `430 Гц`;
- профиль `250 Гц` все еще нарушает условие `gyro_rate >= 1.8 * loop_rate`;
- профиль `100 Гц` тоже допускает взведение, но сильнее снижает частоту
  главного цикла без дополнительной инженерной необходимости.

Таким образом, исправление выполнено минимальным параметрическим
изменением без отключения `ARMING_CHECK`.

## 7. Дополнительная проверка после arm

После выбора профиля `SCHED_LOOP_RATE = 200` выполнены:

- `scripts/run_ardupilot_arm_pwm_response.m`;
- `scripts/analyze_ardupilot_motor_order.m`;
- `scripts/run_ardupilot_first_controlled_response.m`.

Подтверждено:

- команда `arm` принята (`COMMAND_ACK = 0`);
- диапазон `motor_pwm_us` в проверочном прогоне после arm:
  `1000..1100 us`;
- диапазон `motor_cmd_radps`:
  `0..90 рад/с`;
- порядок каналов и двигателей сохраняет прямое соответствие `1..4`;
- первый управляемый расчетный прогон выполнен;
- `valid_rx_count = 507`,
  `response_tx_count = 506`,
  `last_frame_count = 1637`,
  состояние модели остается конечным.

При этом данный прогон не подтверждает устойчивый автоматический полет и
не является летной валидацией.

## 9. Что не является летной валидацией

Результаты TASK-22 не являются:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением пригодности к штатной эксплуатации без дальнейшей
  стендовой отработки.

Подтвержден только инженерный результат:

- блокер предполетной проверки по частоте гироскопа снят при
  `SCHED_LOOP_RATE = 200`;
- команда `arm` принята;
- ненулевой ШИМ и ненулевые команды частоты вращения винтов появились в
  режиме `ArduPilot JSON + MATLAB-модель`.

## 10. Следующий этап

Следующий этап должен быть посвящен уже не предполетной проверке
ArduPilot, а качеству реакции математической модели движения после arm:

- удержанию ненулевого ШИМ в более длинном окне наблюдения;
- анализу роста тяги и вертикального движения после команды взведения;
- дальнейшей проверке контуров управления без заявлений о летной
  валидации.
