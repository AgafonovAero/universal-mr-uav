# TASK-21. Калибровка акселерометра ArduPilot для режима JSON-модели

## Назначение

Документ фиксирует фактическую диагностику отказа взведения
`Arm: 3D Accel calibration needed` для режима:

- внешний комплекс управления полетом `ArduPilot`;
- внешний обмен `JSON/UDP`;
- программно-реализованная математическая модель движения БВС в `MATLAB`.

Цель этапа состояла в том, чтобы не угадывать знак или тип ускорения,
а определить:

1. какие параметры калибровки акселерометра реально загружаются в
   `ArduPilot` после старта;
2. чем JSON-режим отличается от штатного внутреннего `SITL`, где
   взведение проходит;
3. можно ли снять обязательное требование калибровки штатным способом,
   не отключая `ARMING_CHECK`.

## Исходное состояние после TASK-20

До начала TASK-21 было подтверждено:

- штатный `ArduPilot SITL` без внешней модели взводится и выполняет
  команду взлета;
- режим `ArduPilot JSON + MATLAB-модель` сохраняет устойчивый обмен;
- профиль `INS_ENABLE_MASK = 1` устранил отказ
  `Arm: Accels inconsistent`;
- новым блокирующим сообщением стало
  `Arm: 3D Accel calibration needed`.

Таким образом, первичный отказ по рассогласованию нескольких
акселерометров был уже снят, но `ArduPilot` по-прежнему считал, что
обязательная калибровка основного акселерометра не завершена.

## Что означает сообщение `Arm: 3D Accel calibration needed`

Сообщение `Arm: 3D Accel calibration needed` означает, что
`ArduPilot` не считает состояние калибровки акселерометра
достаточным для разрешения взведения.

Для данного стенда это важно по двум причинам:

1. наличие числовых параметров `INS_ACCOFFS_*` и `INS_ACCSCAL_*`
   само по себе не гарантирует, что `ArduPilot` признает калибровку
   завершенной;
2. в JSON-режиме с внешней моделью проверка калибровки выполняется
   так же, как и в штатном режиме, поэтому нельзя считать задачу
   решенной без фактического прохождения этой проверки.

## Диагностика параметров после загрузки

Сценарий `diagnose_ardupilot_accel_calibration_params.m` показал, что
после загрузки профиля TASK-20 в JSON-режиме доступны, в частности,
следующие параметры:

- `INS_ENABLE_MASK = 1`;
- `INS_USE = 1`;
- `INS_USE2 = 0`;
- `INS_USE3 = 0`;
- `INS_ACC_ID = 2753028`;
- `INS_ACCOFFS_X/Y/Z = 0.001`;
- `INS_ACCSCAL_X/Y/Z = 1.001`.

При этом часть ожидавшихся имен в текущей конфигурации параметров не
вернулась в точном виде:

- `AHRS_ORIENTATION`;
- `ARMING_CHECK`;
- `LOG_DISARMED`.

Сравнение со штатным внутренним `SITL` показало:

- в штатном режиме `INS_ENABLE_MASK = 127`;
- в штатном режиме `INS_USE2 = 1`;
- в штатном режиме активен второй экземпляр акселерометра
  `INS_ACC2_ID = 2753036`;
- в JSON-режиме после профиля TASK-20 второй экземпляр отключен
  (`INS_USE2 = 0`, `INS_ACC2_ID = 0`).

При этом значения калибровочных коэффициентов основного акселерометра
между штатным режимом и JSON-режимом практически совпадают.

Инженерный вывод:

- отказ `3D Accel calibration needed` не объясняется простой потерей
  числовых коэффициентов `INS_ACCOFFS_*` и `INS_ACCSCAL_*`;
- требуется именно штатное подтверждение калибровки в терминах
  `ArduPilot`.

## Штатная простая калибровка

Были проверены три пути штатной простой калибровки:

1. `MAVProxy accelcalsimple`;
2. `MAVProxy ground`;
3. `MAVLink MAV_CMD_PREFLIGHT_CALIBRATION` с параметром простой
   калибровки.

Фактически сработал третий путь:

- выбран метод `mavlink_simple`;
- `COMMAND_ACK = 0`;
- устойчивый обмен при этом не потерян.

Параметрический профиль `task21_single_accel_calibrated.parm` в рамках
TASK-21 не потребовался, поскольку штатная простая калибровка
выполнилась и сняла именно сообщение
`Arm: 3D Accel calibration needed`.

## Повторная проверка взведения

После простой калибровки был выполнен повторный прогон
`ArduPilot JSON + MATLAB-модель`.

Фактический результат:

- `valid_rx_count = 502`;
- `response_tx_count = 501`;
- `last_frame_count = 989`;
- `COMMAND_ACK = 4`;
- взведение не выполнено;
- новое первое сообщение отказа:
  `Arm: Gyro 0 rate 478Hz < loop rate*1.8 720Hz`.

Диапазоны команд винтомоторной группы остались:

- `motor_pwm_us = 1000..1000 us`;
- `motor_cmd_radps = 0..0 рад/с`.

Следовательно, TASK-21 подтвердил два факта:

1. отказ `3D Accel calibration needed` действительно снят;
2. следующий блокер относится уже не к калибровке акселерометра, а к
   частоте контура и частоте данных гироскопа.

## Что не является летной валидацией

Полученный результат не является:

- летной валидацией;
- подтверждением устойчивого автоматического полета;
- подтверждением работоспособности замкнутого контура траекторного
  управления.

На текущем этапе подтверждено только следующее:

- устойчивый обмен `ArduPilot JSON/UDP` с внешней `MATLAB`-моделью
  сохранен;
- отказ по обязательной калибровке акселерометра устранен штатным
  способом;
- следующий блокирующий фактор локализован и сформулирован явно.

## Следующий этап

Следующим этапом следует считать отдельную диагностику и устранение
отказа по частоте гироскопа и частоте главного цикла:

- проверить фактическую частоту выдачи инерциальных данных в JSON-режиме;
- сопоставить ее с требованиями `ArduPilot` по `Gyro rate`;
- выбрать минимальное параметрическое или исполнительное решение без
  отключения `ARMING_CHECK`.
