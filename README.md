# Universal MR UAV

`universal-mr-uav` представляет собой универсальную математическую и
имитационную модель беспилотного воздушного судна мультироторного типа.
Репозиторий предназначен для разработки, верификации и поэтапного
расширения расчетного ядра в среде MATLAB/Simulink без переноса
физической логики в бинарные схемы.

Проект ориентирован на задачи, характерные для беспилотной авиационной
системы:

- построение математической модели движения;
- моделирование винтомоторной группы;
- формирование измерений бортовых датчиков;
- оценивание параметров движения и ориентации;
- подготовка средств сопряжения с внешними комплексами управления
  полетом;
- верификация на уровне MIL и SIL.

При первом употреблении сокращений в проекте принимаются следующие
пояснения:

- MIL — моделирование с математической моделью объекта управления;
- SIL — программное моделирование с внешним программным комплексом
  управления.

## Назначение репозитория

Репозиторий служит воспроизводимой инженерной основой для исследований и
разработки алгоритмов управления мультироторным беспилотным воздушным
судном.

На текущем этапе проект решает следующие практические задачи:

- хранит основное инженерное описание объекта управления в текстовом
  `.m`-коде;
- поддерживает воспроизводимые локальные проверки;
- отделяет физику объекта управления от тонких имитационных оболочек;
- готовит сопряжение с внешними программными комплексами управления,
  включая `ArduPilot` и `PX4`.

## Главный архитектурный принцип

Исходное достоверное описание модели должно находиться в текстовых
файлах репозитория, а именно:

- в функциях MATLAB, содержащих математическое описание движения;
- в явных конфигурациях и наборах параметров;
- в документации проекта;
- в исходных журналах прогонов;
- в воспроизводимых табличных и графических артефактах.

`Simulink` допускается только как тонкая имитационная оболочка,
построенная над уже существующим расчетным ядром. Ручной перенос
физических соотношений в блок-схемы считается архитектурной ошибкой.

## Текущая стадия

Репозиторий находится на стадии `Stage-1.5+`.

На этой стадии уже присутствуют:

- расчетное ядро математической модели движения;
- подсистема датчиков в `.m`-коде;
- алгоритм оценивания состояния в `.m`-коде;
- тонкая оболочка MIL-уровня;
- подготовительные средства SIL-уровня;
- сценарии замкнутого моделирования с использованием оцененных
  параметров состояния;
- заготовка средства сопряжения с `ArduPilot` для программного
  моделирования без реального сетевого обмена.

## Структура проекта

Основные каталоги репозитория имеют следующее назначение:

- `src/+uav/core/` — кинематика, динамика, кватернионы и базовые
  состояния объекта управления;
- `src/+uav/vmg/` — модель винтомоторной группы;
- `src/+uav/env/` — модель внешней среды;
- `src/+uav/ctrl/` — законы управления и демонстрационные регуляторы;
- `src/+uav/sensors/` — подсистема датчиков;
- `src/+uav/est/` — алгоритм оценивания состояния;
- `src/+uav/sim/` — исполнители сценариев моделирования и вспомогательные
  процедуры;
- `src/+uav/sil/` — средства подготовки к программному моделированию с
  внешним программным комплексом управления;
- `src/+uav/sl/` — обслуживание тонких оболочек `Simulink`;
- `src/+uav/ardupilot/` — средство сопряжения с `ArduPilot` на этапе
  подготовки JSON-обмена.

Служебные каталоги:

- `scripts/` — сценарии локальных запусков и построения оболочек;
- `tests/` — модульные и интеграционные проверки;
- `artifacts/logs/` — исходные журналы прогонов;
- `artifacts/reports/` — отчеты по задачам;
- `docs/` — проектная документация.

## Системы координат и единицы измерения

В расчетном ядре приняты следующие соглашения:

- земная система координат — `NED`;
- связанная система координат — оси `X` вперед, `Y` вправо, `Z` вниз;
- внутреннее представление ориентации — кватернион;
- углы Эйлера используются только для отображения, отладки и отдельных
  интерфейсных задач;
- все величины задаются в системе СИ;
- углы в коде записываются в радианах, а переход к градусам выполняется
  только явно.

## Что реализовано в расчетном ядре

На текущем этапе в проекте реализованы:

- жесткотельная математическая модель движения;
- простая модель тяги и реактивного момента винтомоторной группы вида
  `T = kT * omega^2` и `Q = kQ * omega^2`;
- квадрокоптерная схема смешения `quad-X`;
- демонстрационные регуляторы для базовых сценариев;
- явная подсистема датчиков;
- алгоритм оценивания состояния с кватернионным описанием ориентации;
- сценарии взлета, зависания и ступенчатых воздействий;
- заготовки средств сопряжения с внешними программными комплексами
  управления.

## Роль MATLAB и Simulink

`MATLAB` используется как основная среда вычислений и хранения
инженерной логики.

`Simulink` применяется только там, где необходима тонкая имитационная
оболочка для оркестрации уже существующих программных интерфейсов.

Это означает, что:

- физика объекта управления не переносится в ручные блок-схемы;
- подсистема датчиков и алгоритм оценивания состояния остаются в
  текстовом коде;
- двоичные модели не становятся главной точкой сопровождения проекта.

## Средства сопряжения с внешними комплексами управления

Репозиторий развивает несколько направлений сопряжения с внешними
комплексами управления полетом.

В настоящее время в проекте существуют:

- подготовительные средства для программного моделирования;
- исследовательская ветка `task/08a-atc-mil-bridge`, связанная с внешним
  контроллером `atc_controller`;
- TASK-10, посвященная подготовке средства сопряжения с `ArduPilot`;
- TASK-11, посвященная первому воспроизводимому обмену данными с
  `ArduPilot` по `JSON` и `UDP`.

Важно подчеркнуть:

- ветка PR `#7` не является зависимостью TASK-10;
- TASK-10 не вводит реальный сетевой обмен с работающим `ArduPilot SITL`;
- TASK-11 вводит только первый уровень обмена данными и не подтверждает
  устойчивый автоматический полет под управлением внешнего комплекса;
- ветка PR `#7` остается отдельным исследовательским направлением и не
  включается в состав TASK-10 и TASK-11.

## Документация проекта

сопряжению и отчетам по задачам.

К числу ключевых материалов относятся:

- документ по сопряжению с `ArduPilot` для программного моделирования;
- документ по обмену данными с `ArduPilot` по `JSON` и `UDP`;

Все пояснительные материалы по проекту должны быть написаны на русском
техническом языке с использованием нормативной авиационной и
динамической терминологии.

## Локальная верификация

Каждая инженерная задача считается завершенной только при наличии:

- обновленного или созданного исходного кода;
- проверок в каталоге `tests/`;
- сценариев локального запуска;
- сохраненных исходных журналов прогонов;
- отчета о результатах, ограничениях и следующем этапе работ.

Минимальный цикл локальной проверки:

```matlab
scripts/bootstrap_project.m
runtests('tests')
```

При необходимости дополнительно запускаются специализированные сценарии
конкретной задачи.

В рамках TASK-09 выполнен переход к замкнутым сценариям моделирования,
где демонстрационные регуляторы используют результаты алгоритма

Этот этап подтвердил:

- сохранение основной физики в расчетном ядре;
- готовность репозитория к дальнейшему расширению средств сопряжения.

## Состояние TASK-10

В рамках TASK-10 подготовлено средство сопряжения с `ArduPilot` для
будущего JSON-обмена данными при программном моделировании.

На текущем этапе реализованы:

- типовая конфигурация каналов обмена;
- проверка локальной готовности вычислительной среды;
- формирование пакета данных состояния и измерений;
- разбор команд широтно-импульсной модуляции;
- преобразование длительности импульсов в команды частоты вращения
  винтов;
- проверочные замкнутые прогоны без запуска реального внешнего
  автопилота;
- документирование ограничений текущего этапа.

TASK-10 не является летной валидацией и не подтверждает готовность к
полноценному взаимодействию с работающим `ArduPilot SITL`.

## Состояние TASK-11

В рамках TASK-11 средство сопряжения с `ArduPilot` расширено до первого
воспроизводимого обмена данными по `JSON` и `UDP`.

На текущем этапе реализованы:

- кодирование расчетного пакета данных в строку `JSON` для передачи во
  внешний комплекс управления;
- разбор двоичного выходного пакета `ArduPilot SITL` с поддержкой
  вариантов на 16 и 32 канала;
- транспортный уровень `UDP` в среде MATLAB с открытием, пошаговым
  приемом, передачей и корректным закрытием средства обмена;
- расчетный сценарий обмена, который использует реальный ответ
  `ArduPilot`, если пакет действительно получен;
- отдельный сценарий самопроверки без `ArduPilot`, подтверждающий
  корректность кодирования `JSON`, разбора двоичного пакета и
  преобразования длительности импульсов ШИМ в команды частоты вращения
  винтов;
- сценарий краткой попытки обмена с уже запущенным `ArduPilot SITL`,
  который честно фиксирует отсутствие ответа, если среда не готова.

TASK-11 не является:

- летной валидацией;
- подтверждением штатного автоматического полета;
- завершенным согласованием всех режимов работы внешнего комплекса
  управления;
- заменой следующего этапа, на котором потребуется настроить рабочую
  среду `WSL`, запуск `sim_vehicle.py` и согласование параметров
  конкретного объекта управления.

## Ограничения текущего этапа

На текущей стадии проект имеет следующие ограничения:

- отсутствует подтвержденный устойчивый обмен с работающим
  `ArduPilot SITL` на полном сценарии автоматического полета;
- отсутствует настройка внешнего комплекса управления под параметры
  конкретного объекта управления;
- `Simulink` применяется только как тонкая оболочка над расчетным ядром;
- необходимы последующие этапы верификации и согласования интерфейсов.

## Ближайшие этапы работ

Приоритетные направления дальнейшего развития:

1. подготовка локальной среды для реального `ArduPilot SITL`;
2. согласование порядка двигателей, режимов и диапазонов ШИМ;
3. подтверждение устойчивого обмена по `UDP` с работающим
   `ArduPilot SITL`;
4. настройка обмена данными с внешним комплексом управления на
   длительном расчетном прогоне;
5. расширение сценариев программного моделирования;
6. дальнейшее развитие направлений `ArduPilot SIL`, `PX4 SIL` и
   `PX4 HIL`.
