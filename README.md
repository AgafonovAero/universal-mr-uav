# Universal MR UAV

`universal-mr-uav` представляет собой универсальную математическую и
имитационную модель беспилотного воздушного судна мультироторного типа.
Репозиторий предназначен для разработки, верификации и поэтапного
расширения расчетного ядра в среде MATLAB/Simulink без переноса
физической логики в бинарные схемы.

Проект ориентирован на задачи, характерные для беспилотной авиационной
системы:

- построение математической модели движения;
- моделирование винтомоторной группы;
- формирование измерений бортовых датчиков;
- оценивание параметров движения и ориентации;
- подготовка средств сопряжения с внешними комплексами управления
  полетом;
- верификация на уровне MIL и SIL.

При первом употреблении сокращений в проекте принимаются следующие
пояснения:

- MIL — моделирование с математической моделью объекта управления;
- SIL — программное моделирование с внешним программным комплексом
  управления.

## Назначение репозитория

Репозиторий служит воспроизводимой инженерной основой для исследований и
разработки алгоритмов управления мультироторным беспилотным воздушным
судном.

На текущем этапе проект решает следующие практические задачи:

- хранит основное инженерное описание объекта управления в текстовом
  `.m`-коде;
- поддерживает воспроизводимые локальные проверки;
- отделяет физику объекта управления от тонких имитационных оболочек;
- готовит сопряжение с внешними программными комплексами управления,
  включая `ArduPilot` и `PX4`.

## Главный архитектурный принцип

Исходное достоверное описание модели должно находиться в текстовых
файлах репозитория, а именно:

- в функциях MATLAB, содержащих математическое описание движения;
- в явных конфигурациях и наборах параметров;
- в документации проекта;
- в исходных журналах прогонов;
- в воспроизводимых табличных и графических артефактах.

`Simulink` допускается только как тонкая имитационная оболочка,
построенная над уже существующим расчетным ядром. Ручной перенос
физических соотношений в блок-схемы считается архитектурной ошибкой.

## Текущая стадия

Репозиторий находится на стадии `Stage-1.5+`.

На этой стадии уже присутствуют:

- расчетное ядро математической модели движения;
- подсистема датчиков в `.m`-коде;
- алгоритм оценивания состояния в `.m`-коде;
- тонкая оболочка MIL-уровня;
- подготовительные средства SIL-уровня;
- сценарии замкнутого моделирования с использованием оцененных
  параметров состояния;
- заготовка средства сопряжения с `ArduPilot` для программного
  моделирования без реального сетевого обмена.

## Структура проекта

Основные каталоги репозитория имеют следующее назначение:

- `src/+uav/core/` — кинематика, динамика, кватернионы и базовые
  состояния объекта управления;
- `src/+uav/vmg/` — модель винтомоторной группы;
- `src/+uav/env/` — модель внешней среды;
- `src/+uav/ctrl/` — законы управления и демонстрационные регуляторы;
- `src/+uav/sensors/` — подсистема датчиков;
- `src/+uav/est/` — алгоритм оценивания состояния;
- `src/+uav/sim/` — исполнители сценариев моделирования и вспомогательные
  процедуры;
- `src/+uav/sil/` — средства подготовки к программному моделированию с
  внешним программным комплексом управления;
- `src/+uav/sl/` — обслуживание тонких оболочек `Simulink`;
- `src/+uav/ardupilot/` — средство сопряжения с `ArduPilot` на этапе
  подготовки JSON-обмена.
- `tools/ardupilot/` — операторские сценарии подготовки Windows/WSL
  и внешнего каталога `ArduPilot`.

Служебные каталоги:

- `scripts/` — сценарии локальных запусков и построения оболочек;
- `tests/` — модульные и интеграционные проверки;
- `artifacts/logs/` — исходные журналы прогонов;
- `artifacts/reports/` — отчеты по задачам;
- `docs/` — проектная документация.

## Системы координат и единицы измерения

В расчетном ядре приняты следующие соглашения:

- земная система координат — `NED`;
- связанная система координат — оси `X` вперед, `Y` вправо, `Z` вниз;
- внутреннее представление ориентации — кватернион;
- углы Эйлера используются только для отображения, отладки и отдельных
  интерфейсных задач;
- все величины задаются в системе СИ;
- углы в коде записываются в радианах, а переход к градусам выполняется
  только явно.

## Что реализовано в расчетном ядре

На текущем этапе в проекте реализованы:

- жесткотельная математическая модель движения;
- простая модель тяги и реактивного момента винтомоторной группы вида
  `T = kT * omega^2` и `Q = kQ * omega^2`;
- квадрокоптерная схема смешения `quad-X`;
- демонстрационные регуляторы для базовых сценариев;
- явная подсистема датчиков;
- алгоритм оценивания состояния с кватернионным описанием ориентации;
- сценарии взлета, зависания и ступенчатых воздействий;
- заготовки средств сопряжения с внешними программными комплексами
  управления.

## Роль MATLAB и Simulink

`MATLAB` используется как основная среда вычислений и хранения
инженерной логики.

`Simulink` применяется только там, где необходима тонкая имитационная
оболочка для оркестрации уже существующих программных интерфейсов.

Это означает, что:

- физика объекта управления не переносится в ручные блок-схемы;
- подсистема датчиков и алгоритм оценивания состояния остаются в
  текстовом коде;
- двоичные модели не становятся главной точкой сопровождения проекта.

## Состояние TASK-14

В рамках TASK-14 проект получает однокнопочный стенд, объединяющий
операторскую проверку среды, подготовку внешнего комплекса управления,
запуск `ArduPilot SITL`, запуск математической модели движения и
диагностику наземной станции управления.

Главной точкой входа служит файл `OneKeyArduPilotStand.m`.
Он является тонким входным сценарием и передает управление в
`scripts/onekey_ardupilot_stand.m`, где размещена основная логика
режимов `check`, `install`, `start`, `stop`, `status` и `full`.

Типовые вызовы выполняются так:

```matlab
OneKeyArduPilotStand("check")
OneKeyArduPilotStand("install")
OneKeyArduPilotStand("start")
OneKeyArduPilotStand("status")
OneKeyArduPilotStand("full")
```

Режим `check` печатает состояние Windows, `MATLAB`, `WSL`, внешнего
каталога `ArduPilot`, `sim_vehicle.py`, `Mission Planner`,
`QGroundControl`, доступности `UDP` и наличия ключевых сценариев
сопряжения.

Режим `install` предназначен для подготовки среды, но по умолчанию
работает как dry-run и не выполняет скрытых административных действий.
Он формирует точные команды для:

- подготовки `WSL`;
- подготовки внешнего каталога `ArduPilot` внутри `WSL`;
- установки `Mission Planner`;
- установки `QGroundControl`.

Режим `start` отвечает за запуск стенда.
Он должен явно печатать:

- какой внешний сценарий запускается;
- какой порт используется для `MAVLink`;
- какой адрес и порт используются для `JSON/UDP`;
- получен ли двоичный пакет от `ArduPilot`;
- выполнена ли ответная передача строки `JSON`;
- запущена ли наземная станция управления.

Режим `stop` предназначен для безопасного завершения только тех
процессов, которые были запущены самим стендом.
Режим `status` печатает сводную картину по среде, портам, процессам и
последним журналам.
Режим `full` объединяет последовательность `check -> install -> start ->
status` и останавливается в точке, где необходимо ручное действие
оператора.

Состав TASK-14 распределен по следующим группам файлов:

- `src/+uav/+setup/` — функции конфигурации и диагностики стенда;
- `tools/ardupilot/windows/` — PowerShell-сценарии проверки, установки,
  запуска и остановки;
- `tools/ardupilot/wsl/` — Bash-сценарии подготовки `ArduPilot` и
  запуска `ArduCopter` в режиме `JSON`;
- `scripts/` — MATLAB-сценарии полного запуска и диагностики наземной
  станции управления.

TASK-14 не отменяет архитектурных ограничений проекта.
На этом этапе по-прежнему не заявляются:

- устойчивый автоматический полет;
- летная валидация;
- подтвержденный двусторонний обмен без фактически принятого двоичного
  пакета от `ArduPilot`;
- подключение наземной станции управления без фактического появления
  потока `MAVLink`.

Подробное описание стенда приведено в документе
`docs/74_onekey_ardupilot_stand_ru.md`.
