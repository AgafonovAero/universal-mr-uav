# Universal MR UAV

`universal-mr-uav` представляет собой универсальную математическую и
имитационную модель беспилотного воздушного судна мультироторного типа,
предназначенную для поэтапного наращивания программно-реализованного
расчетного ядра в среде MATLAB/Simulink.

Проект ориентирован на разработку и верификацию:

- математической модели движения;
- моделей ВМГ;
- подсистемы датчиков;
- алгоритма оценивания состояния;
- средств сопряжения с внешними комплексами управления полетом;
- тонких имитационных оболочек MIL и SIL.

## Назначение проекта

Репозиторий предназначен для построения воспроизводимой и расширяемой
основы беспилотной авиационной системы мультироторного типа.

На текущем этапе проект решает следующие задачи:

- хранит исходное достоверное описание модели в текстовом `.m`-коде и
  документах;
- сохраняет физическую логику объекта управления вне бинарных схем;
- обеспечивает локальную верификацию через сценарии моделирования,
  модульные проверки и исходные журналы прогонов;
- подготавливает средства сопряжения с внешними комплексами управления
  полетом `ArduPilot` и `PX4`.

## Основной архитектурный принцип

Исходное достоверное описание модели должно находиться в:

- `.m`-функциях расчетного ядра;
- текстовых конфигурациях и preset-файлах;
- документации в формате markdown;
- исходных журналах прогонов;
- воспроизводимых табличных и графических артефактах.

Файлы `models/mil_top.slx` и `models/sil_top.slx` допустимы только как
тонкая имитационная оболочка над уже существующим расчетным ядром.

В проекте запрещается:

- переносить физику объекта управления в ручные блок-схемы Simulink;
- использовать бинарные артефакты как основное место хранения модели;
- подменять выполненные прогоны непроверенными заявлениями об успешности.

## Текущая стадия

Репозиторий находится на стадии `Stage-1.5+`.

На этой стадии уже присутствуют:

- расчетное ядро математической модели движения мультироторного объекта
  управления;
- подсистема датчиков в `.m`-коде;
- алгоритм оценивания состояния в `.m`-коде;
- тонкая MIL-оболочка;
- подготовленный интерфейс SIL-уровня;
- сценарии замкнутого моделирования с использованием алгоритма
  оценивания состояния;
- средство сопряжения с `ArduPilot` на уровне заготовки для программного
  моделирования без реального внешнего обмена данными.

## Структура репозитория

Основные каталоги:

- `src/+uav/core/` - кинематика, динамика, кватернионные операции и
  базовые состояния;
- `src/+uav/vmg/` - модель винтомоторной группы;
- `src/+uav/env/` - модель внешней среды;
- `src/+uav/ctrl/` - законы управления и демонстрационные регуляторы;
- `src/+uav/sensors/` - подсистема датчиков;
- `src/+uav/est/` - алгоритм оценивания состояния;
- `src/+uav/sim/` - исполнители сценариев моделирования и вспомогательные
  процедуры;
- `src/+uav/sil/` - подготовка к сопряжению с внешними комплексами
  управления;
- `src/+uav/sl/` - процедуры обслуживания тонких имитационных оболочек
  Simulink;
- `src/+uav/ardupilot/` - средство сопряжения с `ArduPilot` для
  программного моделирования на текущем этапе работ;
- `scripts/` - сценарии запуска, построения оболочек и демонстрационных
  прогонов;
- `tests/` - модульные и интеграционные проверки;
- `artifacts/logs/` - исходные журналы прогонов;
- `artifacts/reports/` - отчеты по задачам;
- `docs/` - проектная документация.

## Системы координат и единицы измерения

В проекте приняты следующие соглашения:

- земная система координат в расчетном ядре - `NED`;
- связанная система координат - оси `X` вперед, `Y` вправо, `Z` вниз;
- внутреннее представление ориентации - кватернион;
- углы Эйлера применяются только для отображения, отладки и отдельных
  интерфейсных задач;
- все расчетные величины задаются в системе СИ;
- углы в коде задаются в радианах, а перевод в градусы выполняется только
  явно.

## Что уже реализовано

На текущем этапе реализованы:

- жесткотельная математическая модель движения;
- простая модель тяги и реактивного момента ВМГ вида
  `T = kT * omega^2`, `Q = kQ * omega^2`;
- квадрокоптерная схема смешения `quad-X`;
- минимальные контуры для демонстрационного управления;
- подсистема датчиков с явным формированием измерений;
- алгоритм оценивания состояния с кватернионным представлением
  ориентации;
- сценарии взлета, зависания и ступенчатых воздействий;
- заготовки средств сопряжения для внешних комплексов управления полетом.

## Средства сопряжения с внешними комплексами управления полетом

Проект поэтапно готовит интеграцию с внешними комплексами управления
полетом, не делая бинарные оболочки основным местом хранения инженерной
логики.

На сегодня в репозитории присутствуют:

- исследовательская ветка `task/08a-atc-mil-bridge` для внешнего
  контроллера `atc_controller`;
- подготовительные средства SIL-уровня;
- заготовка модуля сопряжения с `ArduPilot` через JSON-обмен на уровне
  пакетов данных и проверочных замкнутых прогонов.

Важно:

- ветка PR `#7` не является зависимостью TASK-10;
- TASK-10 не реализует реальный обмен с работающим `ArduPilot SITL`;
- текущие проверочные прогоны служат только для первичной проверки
  работоспособности границы сопряжения.

## Проверки и воспроизводимость

Каждая инженерная задача считается завершенной только при наличии:

- созданных или обновленных файлов исходного кода;
- модульных проверок;
- сценариев локального запуска;
- исходных журналов прогонов;
- отчета о задаче с ограничениями и следующими шагами.

Минимальный типовой цикл локальной проверки:

```matlab
scripts/bootstrap_project.m
runtests('tests')
```

Дополнительные сценарии зависят от конкретной задачи и перечисляются в
отчете соответствующей задачи.

В рамках TASK-09 выполнен переход к замкнутым сценариям моделирования,
где демонстрационные регуляторы используют результаты алгоритма

объекта.

Для TASK-09 поддерживаются:

- `scripts/run_demo_takeoff_to_50m.m`;
- `scripts/run_demo_pitch_step_minus10deg.m`;
- соответствующие сценарии построения графиков;
- журналы прогонов и отчет в `artifacts/reports/task_09_summary_ru.md`.

## TASK-10

В рамках TASK-10 подготовлена заготовка средства сопряжения с
`ArduPilot` для программного моделирования через JSON-интерфейс.

На текущем этапе реализованы:

- типовая конфигурация обмена;
- проверка локальной готовности вычислительной среды;
- формирование пакета данных состояния и измерений;
- разбор команд широтно-импульсной модуляции;
- преобразование команд ШИМ в команды по частоте вращения винтов;
- проверочные замкнутые прогоны без запуска реального внешнего
  автопилота;
- модульные проверки и отчет о задаче.

TASK-10 не является летной валидацией и не подтверждает готовность
репозитория к полноценному взаимодействию с реальным `ArduPilot SITL`.

## Ограничения текущего этапа

На текущей стадии проект имеет следующие ограничения:

- отсутствует полноценный внешний обмен данными с работающим
  `ArduPilot SITL`;
- отсутствует полноценная настройка внешнего комплекса управления
  полетом под параметры объекта управления;
- `Simulink` используется только как тонкая имитационная оболочка;
- архитектура рассчитана на поэтапное расширение и не претендует на
  штатную эксплуатацию без последующих задач верификации и настройки.
